Техническое задание

Escort Platform (Lovnge) — премиум-каталог анкет, CMS-панель, бронирование с эскроу на TON USDT, модерация и ЛК для клиентов/менеджеров/моделей.

Версия: 1.0 Обновлено: 2026-04-19 Фаза: Phase 1 Core CMS ~90% (нед. 14–15) Репо: github.com/kabuto-lab/e1

Этот документ — единый источник правды ТЗ. Связанные документы: ENTITY.md (конституция работы), DESIGN.md (токены), docs/REBUILD_SPEC_EN.md (stack-agnostic спека), apps/web/public/platform-blueprint.html (живой статус/план), FEATURES.md (чек-лист фич).

1. Назначение и бизнес-контекст

Продукт: премиум-платформа агентства сопровождения. Публичный сайт с каталогом анкет + админка CMS + личные кабинеты клиентов, менеджеров и моделей + бронирование через эскроу в TON USDT.

Ключевая ценность: безопасная сделка (эскроу до подтверждения встречи), верифицированные анкеты, скрытие контактов до оплаты, премиальный визуал.

Рынок и язык: RU-first, Data region ru, аудит-логи 1825 дней.

Позиционирование (юридическое): информационный сервис размещения анкет — не работодатель, не оказывает услуги. Рассматриваются ОКВЭД 74.90 / 82.99 / 96.09.

7. RBAC и guards

  • JwtAuthGuard: Bearer из Authorization, кладёт payload в req.user.
  • RolesGuard: декоратор @Roles('admin','manager', ...) сверяется с user.role.
  • JWT claims: sub, email, role, subscriptionTier, jti. Access TTL по умолчанию 15m, refresh — 7d.
  • Секреты: JWT_SECRET (64 hex) ≠ JWT_REFRESH_SECRET (min 32).
  • Публичные контроллеры: помечены @Public() или идут вне защищённого префикса.
  • RateLimitModule: глобально 100 req / 60s, для /auth/login, /auth/register, /contact — строже (отдельные декораторы).

8. Эскроу TON USDT (~90%)

8.1 State machine

booking.status:
  draft
    → pending_payment  (client initiates)
    → escrow_funded    (indexer detected USDT deposit)
    → confirmed        (model confirms in LK)
    → in_progress / completed
    → disputed | cancelled | refunded

escrow.status:
  pending_funding → funded → hold_period → released | refunded

8.2 Архитектура TON-эскроу

  • Value objects: EscrowMemo, TonAddress, CryptoAmount (apps/api/src/escrow/domain/value-objects/)
  • TonHotWalletService: TEP-74 jetton transfers; mnemonic в env (рекомендуется ротация до фиат-объёмов)
  • TonEscrowIndexerService: поллинг tonapi.io, account events, matching по EscrowMemo. Хранит lastLt.
  • TonEscrowDepositGuard: guard на ingest endpoint
  • BookingTonModal (UI): 4 шага — confirm → создание брони+эскроу → инструкции + QR → поллинг → reveal контактов
  • Audit: escrow_audit_events — каждое изменение статуса

8.3 Открытые вопросы (нед. 14–15)

  • e2e тесты эскроу (сейчас только unit на VO и service) — блокер CI
  • Anti-replay на ingest
  • Staleness-метрика индексера TonAPI (возраст lastLt)
  • Rate-limit публичных эскроу-эндпоинтов
  • Ротация mnemonic hot wallet + бэкап-регламент

15. CMS gap-анализ

Проект — это вертикальная специализированная CMS для агентства сопровождения, не general-purpose (не WP/Strapi). Ниже — сравнение возможностей.

Легенда: есть реализовано · частично есть база · нет не реализовано · не нужно вне скоупа

CMS-возможность Статус Что есть сейчас Пригодится / как добавить
Контент-типы (models, bookings, reviews) есть Drizzle-схемы для всех доменных сущностей, CRUD API, дашборд Готово. Плюс vs. WP: жёсткая типизация, миграции, нет plugin-шва
Медиа-библиотека есть MinIO/S3 presigned upload, confirm, moderation_status, sort_order, album_category, is_public_visible ✅ На уровне Directus/Payload. Не хватает: image variants (thumb/medium/large) on-the-fly, EXIF strip, focal point
Визуальный редактор страниц (page builder) нет Только platform_settings JSON (hero slogan, typography, logos) Нужен, если добавлять новые лендинги без деплоя. Минимум: JSON-блоки в settings + рендерер в app/page.tsx. Решения: Puck, GrapesJS, собственный блочный редактор
Меню-редактор нет Навигация хардкодом в components/Navbar.tsx Низкий приоритет — меню стабильно. Можно через platform_settings.data.menu JSON
Таксономии (теги, категории) частично model.psychotype_tags, languages, physical_attributes — массивы/JSON на самой сущности Если потребуются глобальные словари → отдельные таблицы tags, tag_assignments
Роли и права (RBAC) есть 4 роли: admin/manager/model/client. Guards + @Roles. JWT claim с role+tier. Для тонкого ACL (per-field / per-row) добавить policy-layer (CASL, casbin)
Модерация контента есть Очередь /dashboard/moderation, pending/approved/rejected на media, reviews, profiles На уровне нормальной CMS. Можно добавить diff-просмотр для правок

Cron / scheduled jobs частично TON-индексер — поллинг (в самом процессе API) Для cleanup sessions / escrow timeouts / notifications batching — @nestjs/schedule

Форма-конструктор не нужно /contact — статическая Не product-critical

Темы / белый-лейбл частично platform_settings: site name, logos, theme tokens (частично) — public API отдаёт безопасный сабсет Для мульти-тенанси потребуется tenant_id на сущностях + per-tenant settings. Сейчас single-tenant
A/B тесты, feature flags нет — Env-based flags на старте; в 2026+ — GrowthBook/Unleash self-host

Импорт каталога (CSV/JSON) нет Только ручной CRUD Скрипт-сидер есть для dev (db:bootstrap). Для прод-импорта — admin-endpoint с валидацией

15.1 Вывод

Платформа уже закрывает ~70–75% стандартных CMS-возможностей, причём в своём вертикальном сегменте — лучше general-purpose CMS: эскроу, RBAC под домен, модерация моделей, state machine бронирования, TON-платежи.

Главные пробелы, которые стоит закрыть до масштабирования:

  1. Бэкапы + cron (pg_dump, cleanup sessions) — критично до prod-траффика
  2. Уведомления (5.13 Telegram + in-app) — UX дыра в эскроу-флоу
  3. Аналитика-дашборд (Metabase/self-host) — поможет операторам
  4. SEO (OG-image, meta) — для роста органики каталога
  5. i18n — если планируется выход за RU-рынок

Не стоит копировать у general-purpose CMS: плагины, маркетплейс, page builder с drag-and-drop — для этого продукта overkill и увеличит attack surface.
