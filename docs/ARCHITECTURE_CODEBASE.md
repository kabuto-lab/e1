# Архитектура кодовой базы (ES / Lovnge)

Документ описывает **потоки данных и ответственность модулей**. Он **не заменяет** построчные комментарии в каждом файле: в монорепозитории тысячи строк; массовое добавление `//` ухудшит читаемость, усложнит ревью и устареет быстрее, чем рефакторинг. Для детализации по файлам используйте сгенерированную **карту** `project-architecture-map.html` + снимок `project-arch-map-data.json`.

---

## 1. Монорепозиторий

| Пакет | Назначение |
|--------|------------|
| `apps/api` | HTTP API (NestJS), бизнес-логика, доступ к БД и MinIO |
| `apps/web` | UI (Next.js App Router), вызовы API через прокси `/api` |
| `packages/db` | Схемы Drizzle, миграции, общие типы таблиц |

Сборка и скрипты оркестрируются **Turborepo** (`turbo.json`, корневой `package.json`).

---

## 2. Инфраструктура разработки

`docker-compose.dev.yml` поднимает **PostgreSQL** (данные), **Redis** (кэш/сессии по мере использования), **MinIO** (объектное хранилище для медиа), **Mailhog** (перехват почты в dev).

API и Web запускаются **отдельными процессами** Node (см. `CLAUDE.md`): API слушает порт **3000**, Next — **3001**.

---

## 3. Слой данных (packages/db)

Drizzle-ORM описывает таблицы в `packages/db/src/schema/*.ts`. Связи и экспорт — через `index.ts` / `relations.ts`.

**Основные сущности:**

- `users` — учётные записи и роли (`admin`, `manager`, `model`, `client`).
- `model_profiles` — публичные/CRM поля карточки модели, slug, публикация.
- `media_files` — файлы в MinIO: ключ, presigned, **видимость**, альбом, порядок, модерация.
- `bookings`, `escrow`, `reviews`, `client_profiles`, `blacklists`, `audit` — доменные таблицы.

**Куда уходят данные:** только через сервисы Nest, которые импортируют Drizzle и выполняют запросы в транзакциях или одиночных операциях.

## 4. API (apps/api)

### 4.1 Запуск и сквозная обработка запроса

1. **`main.ts`** — создание приложения Nest, глобальные фильтры, **Helmet**, **CORS** (белый список origin), **ValidationPipe**, **Swagger** на `/api/docs`.
2. Запрос попадает в **контроллер** модуля.
3. При наличии **`JwtAuthGuard`** — из заголовка `Authorization: Bearer` извлекается JWT, в `request.user` кладётся полезная нагрузка.
4. **`RolesGuard`** сравнивает роль пользователя с `@Roles(...)`.
5. **Сервис** выполняет логику и обращается к БД / MinIO / внешним API.

### 4.2 Модули по доменам

| Модуль | Вход (HTTP) | Выход / побочные эффекты |
|--------|-------------|---------------------------|
| `auth` | `/auth/login`, register, refresh | JWT; запись сессий при необходимости |
| `users` | CRUD пользователей (частично staff-only) | Таблица `users` |
| `models` | CRUD моделей, публичные списки (зависит от guard) | `model_profiles` |
| `profiles` | Расширенные операции CMS, медиа-пайплайн | Профили + MinIO |
| `media` | Presigned upload, обновление видимости | `media_files`, бакет MinIO |
| `bookings` | Создание/смена статусов броней | `bookings` |
| `clients` | CRM по клиентам | `client_profiles` |
| `reviews` | Отзывы | `reviews` |
| `escrow` | Учёт платежей/холда | `escrow` |
| `blacklist` | Чёрные списки | `blacklists` |
| `health` | `/health` | Статус процесса |

**Важно:** часть маршрутов в `models` исторически могла быть без guard — это нужно сверять с прод-политикой (см. роадмэп безопасности).

### 4.3 Безопасность и вспомогательные сервисы

- **`security/helmet.config.ts`**, **`rate-limit.config.ts`** — заголовки и лимиты.
- **`audit-logger.service.ts`** — события для соответствия/отладки.
- **`anti-leak.service.ts`** — логика против утечки контактов в сообщениях (используется там, где подключён).

## 5. Web (apps/web)

### 5.1 Маршрутизация и оболочка

- **`app/layout.tsx`** — корневой layout, шрифты, метаданные.
- **`app/dashboard/layout.tsx`** — боковое меню CMS, отладчик.

### 5.2 Вызовы API

- **`lib/api-url.ts`** — базовый URL: в браузере часто относительный `/api`, на сервере — прямой к бэкенду.
- **`lib/api-client.ts`** — обёртка `fetch` с `Authorization` из `localStorage`, методы под сущности (модели, медиа и т.д.).

**Прокси:** `next.config.js` переписывает `/api/*` на `http://127.0.0.1:3000/*`, чтобы в dev не было CORS с браузера на другой порт.

### 5.3 Публичная часть и дашборд

- **`app/models/*`** — каталог и страница модели; данные с API; медиа фильтруются по `isPublicVisible` на клиенте/сервере в зависимости от страницы.
- **`app/dashboard/models/*`**, **`media`**, **`moderation`**, **`bookings`** — CMS; часть экранов ещё на моках (см. аудит).

## 6. Типичные потоки данных

### Логин

1. Пользователь отправляет email/password на **`POST /auth/login`**.
2. API проверяет хеш, возвращает access/refresh токены.
3. Web сохраняет токены в `localStorage`, далее шлёт `Authorization` на защищённые маршруты.

### Сохранение карточки модели

1. Форма в Next вызывает **`api-client`** → `PUT`/`PATCH` на соответствующий эндпоинт `models` или `profiles`.
2. Сервис валидирует DTO, обновляет строки в **`model_profiles`** (и связанные таблицы при необходимости).

### Загрузка фото

1. Клиент запрашивает **presigned URL** у `media`.
2. Загружает файл напрямую в **MinIO**.
3. Сохраняет метаданные в **`media_files`** (видимость, альбом, порядок).

## 7. Визуальная карта и снимок кода

- **`public/project-architecture-map.html`** — node-based карта; клик по названию узла открывает **off-canvas** с текстом из снимка.

Снимок **ограничен по размеру файла** (см. `MAX_BYTES` в скрипте); большие файлы обрезаются — открывайте полный исходник в IDE.

## 8. Почему не «прокомментирован весь проект»

1. **Объём** — десятки тысяч строк; комментарии дублируют то, что уже выражено типами и именами.
2. **Устаревание** — неверный комментарий хуже отсутствующего.
3. **Точка правды** — Swagger для API, этот документ и карта для структуры, `DESIGN.md` для UI.

Рекомендуется: краткие **JSDoc** на публичных методах сервисов, **README** в неочевидных пакетах, и **issue/ADR** для нетривиальных решений.
