1. Цели, термины, роли

Цель: описать единую систему личных кабинетов для информационной платформы с каталогом анкет, безопасной сделкой (эскроу), закрытым клубом и CRM — с разграничением доступа по ролям.

Термин «личный кабинет» (ЛК): закрытая зона после аутентификации, где пользователь управляет своими данными, заявками, оплатами и коммуникациями в рамках политики платформы.

Роли (канон для ТЗ):

РольНазначение
КлиентГость платформы после регистрации/авторизации; бронь, оплата, избранное, клуб, отзывы.
Модель (исполнитель)Владелец анкеты; медиа, доступность, верификация, статистика, выплаты.
МенеджерВедение анкет «своих» моделей, заявки, связь с клиентами, отчёты.
АдминистраторПользователи, модерация, ЧС, финансы, настройки платформы.

2. Стек и текущая реализация в репозитории

Актуальный стек монорепозитория (см. ENTITY.md, CLAUDE.md): NestJS API, Next.js App Router (фронт), Drizzle ORM, PostgreSQL, MinIO (медиа), Redis при необходимости. Исторический фрагмент исходного бизнес-ТЗ упоминал WordPress/MySQL — для разработки в e1 ориентир: текущий стек, не WP.

Уже есть в коде (частично закрывает ЛК персонала):

  • Маршрут /login, /admin-login; после входа — /dashboard.
  • В UI бокового меню используются подписи «Дэшборд», «Вход» (не формулировка «личный кабинет»).
  • Разделы: главная дашборда, модели (список, создание, редактирование, фото, просмотр), медиатека, бронирования (мок/заготовка), модерация отзывов, настройки, управление главным слайдером (дашборд home).
  • JWT, роли в токене; публичный каталог /models, карточка /models/[slug].

Пробелы относительно полного ТЗ: отдельный полноценный ЛК клиента (избранное, история встреч, эскроу из UI клиента, чат) в основном не реализован как выделенный продукт; отдельный ЛК модели «как у клиента, но для исполнителя» может частично пересекаться с дашбордом при привязке пользователя к анкете.

3. ЛК клиента

Сведение требований из FEATURES.md и раздела 3.1 плана «Тех задание для разраба».

3.1. Функциональные требования

  • Просмотр каталога / карточек моделей (в т.ч. из ЛК deeplink).
  • Запрос на встречу (бронирование); связь с CRM-статусами: запрос → подтверждение → встреча → закрыта.
  • Оплата через механизм эскроу; просмотр статуса платежа и подтверждение успешности встречи (по правилам платформы).
  • История встреч и заявок.
  • Избранные анкеты.
  • Закрытый клуб / VIP / программа лояльности: статус, бонусы, приоритет.
  • Связь с менеджером (модели или платформы): сообщения; опционально файлы.
  • Настройки уведомлений (каналы, типы событий).
  • Загрузка документов при необходимости верификации клиента (если политикой предусмотрено).
  • Уведомления: новые статусы моделей из избранного, новые отзывы, при необходимости — переход к экрану детализации.
  • VIP-группа / эксклюзивные предложения (по эскизу ТЗ).

3.2. Сценарии и навигация (из ТЗ-кнопок)

  • Путь: профиль модели → ЛК клиента → избранное.
  • Путь: ЛК клиента → настройки уведомлений.
  • Путь: ЛК клиента → чат с менеджером.
  • Путь: ЛК клиента → закрытая VIP-группа.
  • Путь: ЛК клиента → подтверждение оплаты / статус эскроу.
  • Путь: ЛК клиента → новые отзывы / лояльность.
  • Выход из сессии с редиректом на публичную главную или экран входа.

4. ЛК модели (исполнителя)

Разделы 3.2 исходного ТЗ и блок «Исполнитель» в FEATURES.md.

  • Просмотр и редактирование анкеты (в пределах прав и модерации).
  • Управление медиа: загрузка, порядок, видимость на публичной карточке (по мере внедрения per-file visibility).
  • Статистика встреч, отмен, показателей надёжности (как в продуктовой концепции).
  • Статус доступности / live-статус («в смене», «свободна» и т.д.).
  • Загрузка документов и прохождение верификации (видео, паспорт — по регламенту).
  • Выплаты и начисления через безопасную сделку; просмотр комиссий.
  • Отзывы по своим анкетам.
  • Расширенные сценарии ТЗ: страница верификации, бонусы модели, уведомления об обновлении статуса.

5. ЛК менеджера и CRM-функции

Разделы «Личный кабинет менеджера» (план), 3.3 ТЗ, CRM в FEATURES.md.

5.1. Обязательный функционал ЛК менеджера

  • Создание и ведение анкет моделей.
  • Просмотр и обработка заявок клиентов по «своим» анкетам.
  • Статистика по встречам, отменам, конверсии (объём детализации — уточняется).
  • Начисления и контроль сделок через эскроу.
  • Работа с отзывами (в рамках выданных прав).
  • Пул / очередь заявок при необходимости.

5.2. CRM (связано с ЛК)

  • Все заявки фиксируются в системе; автопривязка номера/идентификатора клиента.
  • Уведомления ответственным (в т.ч. Telegram владельцу — по интеграции).
  • Статусы сделки: запрос, подтверждение, встреча, закрыта (+ расширения из FEATURES: отказ, не отвечает, ЧС).
  • Карточка заявки: источник (сайт, TG, WA), текст, ответственный, таймлайн.

5.3. Чёрные списки (часть доступна менеджеру/админу по матице)

  • Внутренний ЧС анкет: причины, статус карточки (заблокирована / на проверке / восстановлена), анкета не удаляется физически.
  • ЧС клиентов: «нежелательные клиенты», причины; индикация на карточке анкеты, видит ли модель/менеджер флаг ЧС клиента.

6. ЛК / панель администратора

  • Полный доступ к разделам платформы и данным в рамках политики безопасности.
  • Управление пользователями и ролями.
  • Модерация анкет и контента, восстановление из ЧС.
  • Модерация отзывов (централизованно).
  • Финансовая сводка, комиссии, отчёты (по мере готовности модуля оплат).
  • Настройки платформы, интеграций, лимитов.
  • Просмотр агрегированной статистики.

8. Нефункциональные требования

  • Безопасность: HTTPS, защита персональных данных, аудит критичных действий, RBAC по ролям JWT.
  • Конфиденциальность: политики хранения документов и медиа; минимизация утечек контактов (анти-слив из FEATURES).
  • Юридическая рамка: позиционирование платформы как информационной; тексты согласовать с юристом.
  • Адаптивность: корректная работа ЛК на мобильных, планшетах и десктопах (ориентир 320–1920 px).
  • Дизайн: премиальная эстетика; для публичной части и согласованных экранов — DESIGN.md; дашборд может использовать тему wp-admin.
  • Надёжность API: связность БД с кодом (миграции при деплое); мониторинг 5xx; см. регламент vps:after-pull.

9. Интеграции, затрагивающие ЛК

  • Платежи и эскроу: подтверждение оплаты, вебхуки, возвраты (см. apps/web/public/escrow.html как внутренний конспект).
  • Telegram: уведомления, deep link из ЛК для привязки аккаунта к боту (по готовности).
  • WhatsApp Business API: CRM-канал (по дорожной карте).
  • Верификация: внешние сервисы или ручная модерация — экраны в ЛК модели/админа.
