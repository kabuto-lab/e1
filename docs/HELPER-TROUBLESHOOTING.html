1. Docker / Postgres

Q docker compose команда отвечает «error during connect: ... The system cannot find the file specified».

Причина: не запущен Docker Desktop (Windows).

Решение: открой Docker Desktop (C:\Program Files\Docker\Docker) вручную, дождись «Engine running», потом повтори:

docker-compose -f docker-compose.dev.yml up -d

Проверить статус движка: docker version. Если висит — перезагрузить Docker Desktop.

Q Error response from daemon: driver failed programming external connectivity: Bind for 0.0.0.0:5432 failed: port is already allocated

Причина: на Windows уже стоит нативный PostgreSQL (служба postgresql-x64-17 или 18), он держит 5432.

Решение: остановить и отключить службу Windows:

sc stop postgresql-x64-18
sc config postgresql-x64-18 start= disabled
docker-compose -f docker-compose.dev.yml up -d postgres

Имя службы смотри в Win+R → services.msc. После фикса 5432 будет у Docker-контейнера escort-postgres.

Q API стартует, но отдаёт 503 с текстом «password authentication failed for user "postgres"» (ошибка 28P01). Каталог /models и дашборд оба красные.

Причина: пароль в Docker-томе Postgres не совпадает с паролем в DATABASE_URL. Пароль в томе устанавливается только при первом создании volume; смена POSTGRES_PASSWORD в .env его не обновляет.

Решение (автоматическое):

npm run check:postgres-env   # покажет разницу
npm run ensure:database      # при Docker-контейнере escort-postgres
                             # сам сделает ALTER USER под .env

Решение (ручное, если Postgres не в Docker):

psql -U postgres -c "ALTER USER postgres PASSWORD 'новый-пароль-из-env';"

Никогда: docker compose down -v на проде — сотрёт том и все данные.

На VPS: cd ~/e1 && npm run ensure:database && npm run pm2:reload-api.

Q Поменял POSTGRES_PASSWORD в .env, перезапустил compose — пароль всё равно старый.

Причина: compose использует существующий volume postgres_data, инициализация пароля не запускается повторно.

Решение (dev, данные не жалко):

docker-compose -f docker-compose.dev.yml down -v
docker-compose -f docker-compose.dev.yml up -d

Решение (prod, данные нельзя терять): ALTER USER внутри работающего контейнера под новый пароль из .env, затем pm2:reload-api.

Q MinIO bucket не создаётся, API ругается «NoSuchBucket».

Причина: init-контейнер escort-minio-init не отработал (упал на сети).

Решение:

docker-compose -f docker-compose.dev.yml up -d minio-init

Или вручную в MinIO Console http://localhost:9001 (логин companion_minio_admin / companion_minio_password) → создать bucket escort-media → anonymous download policy.

2. Миграции Drizzle

Q npm run db:generate пишет «No schema changes, nothing to migrate 😴», но я точно добавил колонку.

Причина: drizzle-kit сравнивает с последним снимком packages/db/drizzle/meta/NNNN_snapshot.json. Если снимок уже включает твои изменения (например, предыдущий прогон успел записать снимок, но SQL не вернул) — диффа нет.

Проверка:

ls packages/db/drizzle/meta/

Если есть свежий NNNN_snapshot.json и нет соответствующего NNNN_*.sql — снимок пере-опередил.

Решение:

  1. Открыть последний snapshot, проверить что твои колонки там есть
  2. Написать SQL-миграцию руками под существующий tag из meta/_journal.json
  3. Не делать generate повторно — он не поможет

Q При применении миграции: column "subscription_tier" of relation "users" already exists.

Причина: drizzle-kit сгенерировал «широкий» diff, т.к. в репо отсутствуют снимки промежуточных миграций (0002..0010 были написаны руками без обновления meta). Миграция пытается применить то, что уже есть.

Решение: открыть сгенерированный SQL, оставить только новые для этой итерации операции. Остальное удалить. См. структуру 0011_thankful_kree.sql как образец чистой миграции.

Долгосрочный фикс — backfill снимков meta/0002..0010_snapshot.json. Пока — ручная зачистка после каждого generate.

Q CHECK constraint users_staff_credentials_check не создаётся: «check constraint violated by existing row».

Причина: в таблице users есть staff-row (admin/manager/moderator) без email/password.

Проверка:

psql $DATABASE_URL -c "SELECT id, role, email_hash IS NULL AS no_email, password_hash IS NULL AS no_pw FROM users WHERE role IN ('admin','manager','moderator') AND (email_hash IS NULL OR password_hash IS NULL);"

Решение: заполнить пустые поля на конкретных rows, потом перезапустить миграцию. Или удалить «мусорные» записи, если это тестовые аккаунты.

Q Ошибка «column does not exist» на API после деплоя, хотя локально всё работает.

Причина: на VPS не прогнались миграции. db:migrate пропущен в пайплайне, или упал, или база под другим URL.

Решение на VPS:

cd ~/e1
psql $DATABASE_URL -c "SELECT count(*) FROM drizzle.__drizzle_migrations;"
# сравни количество с локальным; если меньше — прогнать:
npm run db:migrate
npm run pm2:reload-api

3. VPS / PM2 / деплой

Q Поменял .env на VPS, сделал pm2 restart escort-api — API всё равно с прежним DATABASE_URL / секретом.

Причина: pm2 restart не перечитывает ecosystem.config.cjs и сохраняет старый process.env.

Решение:

cd ~/e1 && npm run pm2:reload-api

Эквивалент: pm2 startOrReload ecosystem.config.cjs --only escort-api --update-env. Флаг --update-env критичен.

Для web — npm run pm2:reload-web. Оба есть в vps:after-pull.

Q vps:after-pull падает посередине. Что делать?

Диагностика:

  • npm ci упал — проверь Node ≥22: node -v; диск: df -h
  • npm run build упал — обычно TS-ошибка; читай лог до конца
  • check:postgres-env упал — пароли разошлись, см. секцию Docker
  • db:migrate упал — см. секцию «Миграции»
  • pm2:reload-* упал — посмотри pm2 logs escort-api --lines 50

Частая причина: неполный git pull — локально забыли запушить. Проверь коммит на сервере:

git log --oneline -3

Q pm2 logs escort-api показывает «EADDRINUSE :3000».

Причина: старый процесс не умер (осиротел после неудачного reload).

Решение:

pm2 delete escort-api
pm2 start ecosystem.config.cjs --only escort-api
# или:
npm run pm2:hard-reload-api

Если порт занят посторонним процессом — lsof -i :3000 покажет PID.

4. Telegram бот

Q Бот в логах пишет «Conflict: terminated by other getUpdates request» или «HTTP 409».

Причина: тот же bot token используется одновременно на двух машинах (локально + VPS, или два запущенных процесса PM2).

Решение: использовать два разных бота:

  • Локально: @lovnge_dev_bot, токен в локальном .env
  • VPS: @lovnge_bot, токен только в ~/e1/.env

Быстрая проверка на VPS:

pm2 list | grep escort
# должен быть один процесс с escort-api

Q POST /auth/telegram/login возвращает 401 «invalid hash» при правильных параметрах.

Причины (по убыванию вероятности):

  1. TELEGRAM_BOT_TOKEN в .env — от другого бота (dev vs prod)
  2. auth_date старше 60 секунд — UI медленно отправил
  3. В payload подменён один из полей между widget → сервер
  4. Bot token был revoked у @BotFather → нужен новый

Проверка (hex проверка):

echo -n $TELEGRAM_BOT_TOKEN | sha256sum     # сверяем ключ HMAC

Если клиент подгружает свежий auth_date и всё равно 401 — очевидный признак wrong token.

Q Бот не получает сообщения, в логах тишина.

Причины:

  • TELEGRAM_BOT_ENABLED=false в .env — бот не стартует (master switch)
  • Polling не запустился из-за ошибки в OnModuleInit — pm2 logs escort-api покажет
  • Интернет на VPS блокирует api.telegram.org — curl -sI https://api.telegram.org

Проверка, видит ли TG твой бот:

curl "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/getMe"

Ответ должен быть {"ok":true,"result":{"id":...}}. Иначе — токен битый.

Q Бот роняет весь API, когда Telegram API-сервера недоступны.

Причина: необработанный reject в polling-loop.

Что должно быть в коде:

  • try/catch в OnModuleInit — бот стартует опционально
  • Telegraf .catch() handler
  • Health-endpoint показывает telegram: degraded отдельно от db: ok

Это не должно тащить за собой рестарт API.

Q Deep-link из ЛК не открывается в Telegram (ошибка «bot not found»).

Причины:

  • В ссылке t.me/неправильный_username?start=... — опечатка в TELEGRAM_BOT_USERNAME в .env
  • Бот ещё не имеет публичного username (у @BotFather не задан)

Проверка: открой в браузере t.me/твой_bot_username — должна появиться страница бота. Если нет — проблема в имени.

5. Build / запуск API и Web

Q API при старте падает: Cannot find module '@escort/db'.

Причина (dev): ts-node + tsconfig-paths не резолвит workspace. Обычно означает, что не поставили deps в корне.

Причина (prod): пакет @escort/db указан как main: ./dist/index.js — значит нужен build:

npm run build --workspace=@escort/db

Решение в корне:

npm ci
npm run build

Q Next.js ругается «Error: Hydration failed» на странице с Telegram Login Widget.

Причина: widget подгружается через внешний <script>, на SSR отдаёт другой HTML.

Решение: компонент с widget — 'use client', монтировать только в useEffect, на SSR отдавать плейсхолдер. Либо вообще не использовать widget и пойти через deep-link flow (см. план L2 §Q2).

Q Запуск API в dev даёт «Error: Zod validation: JWT_SECRET expected exactly 64 hex chars».

Причина: в .env секрет короче/длиннее или не hex.

Решение:

openssl rand -hex 32         # сгенерировать 64-hex секрет

Вставить в .env без кавычек. Ещё раз проверить JWT_REFRESH_SECRET (min 32 chars, но не обязан быть hex).

Q npm run dev:apps стартует только API, web не поднимается.

Причина: порт 3001 занят (прежний next dev не умер).

Решение:

powershell scripts/kill-ports.ps1   # если существует
# или вручную:
netstat -ano | findstr :3001
taskkill /F /PID <pid>

6. JWT / авторизация

Q Пользователь внезапно разлогинился через 15 минут, на любой странице 401.

Причина: access token TTL = JWT_ACCESS_EXPIRATION=15m (дефолт). Refresh-rotation во фронте не срабатывает.

Проверка: открой DevTools → Network → смотри, вызывается ли POST /auth/refresh автоматически на 401.

Решение: в apps/web/lib/api-client должен быть interceptor, который на 401 дёргает /auth/refresh и ретраит запрос. Если нет — написать.

Q После рестарта API все refresh-токены слетели.

Причина: JWT секреты изменились в .env (или сгенерированы заново на деплое).

Решение:

  • В dev: не генерировать JWT_SECRET каждый раз — хранить в .env один раз
  • В prod: секрет должен быть зафиксирован и не меняться между деплоями
  • Если пришлось сменить — все пользователи перелогинятся, это ожидаемо

7. MinIO / медиа

Q Presigned URL для upload возвращает SignatureDoesNotMatch или таймаутит в браузере.

Причина: несоответствие между MINIO_ENDPOINT (внутренний, для API) и MINIO_PRESIGN_ENDPOINT (публичный, из браузера).

Проверка .env:

MINIO_ENDPOINT=localhost:9000
MINIO_PRESIGN_ENDPOINT=localhost:9000
MINIO_PUBLIC_URL=http://localhost:9000
NEXT_PUBLIC_MINIO_PUBLIC_URL=http://localhost:9000

На VPS всё это должно указывать на публичный host:port (45.9.40.37:9000 например), не на localhost.

Q Фото загрузилось, но в карточке модели отображается «битая ссылка».

Причина: в БД записан URL с localhost:9000, а смотрим с production-домена.

Решение:

  1. Проверь NEXT_PUBLIC_MINIO_PUBLIC_URL — должен быть публичный
  2. Next.js переписывает ссылки в БД с localhost на публичный URL — убедись что переписывает
  3. Или массовый апдейт в БД: UPDATE media_files SET cdn_url = REPLACE(cdn_url, 'localhost', 'public-host')

Q CORS-ошибка в консоли браузера при upload в MinIO.

Причина: CORS policy на bucket не задан или слетел.

Решение: перезапустить init-контейнер или применить CORS вручную:

mc alias set local http://localhost:9000 companion_minio_admin companion_minio_password
printf '%s' '[{"AllowedOrigins":["*"],"AllowedMethods":["GET","PUT","POST","DELETE","HEAD"],"AllowedHeaders":["*"],"ExposeHeaders":["ETag"]}]' > /tmp/cors.json
mc cors set local/escort-media /tmp/cors.json

8. Bash / tooling

Q В Claude Code CLI команды git или npm run db:generate подвисают без вывода.

Причина: в этом окружении bash-spawn иногда не возвращает EOF для интерактивных процессов (turbo tui, git с pagers). Это особенность терминала, не баг инструментов.

Решение: такие команды запускай в обычном терминале (PowerShell / Windows Terminal), вывод вставляй в чат.

Альтернатива для git — задать неинтерактивный режим:

git --no-pager log --oneline -5
set GIT_PAGER=cat            # PowerShell: $env:GIT_PAGER="cat"

Q При попытке git push — rejected (non-fast-forward).

Причина: в origin/main есть коммиты, которых нет у тебя локально (чаще — с VPS кто-то закоммитил).

Решение:

git fetch origin
git log HEAD..origin/main --oneline     # что там новое
git pull --rebase origin main           # накатить поверх
git push

Не использовать git push --force без причины.

9. CI / тесты

Q GitHub Actions CI падает: «Cannot connect to database».

Причина: unit-тесты пытаются реально ходить в БД, но в .github/workflows/ci.yml нет Postgres service.

Решения:

  • Мокать БД в unit-тестах (правильный путь)
  • Для integration/e2e — добавить Postgres service в ci.yml: 
    services:
  postgres:
    image: postgres:16-alpine
    env:
      POSTGRES_PASSWORD: postgres
    ports: ['5432:5432']

Q Jest в API видит @escort/db, но ругается «Module not found» на соседних импортах.

Причина: в apps/api/package.json jest имеет moduleNameMapper только для @escort/db. Остальные ts-paths не покрыты.

Решение: добавить в jest-конфиг:

"moduleNameMapper": {
  "^@escort/db$": "<rootDir>/../../../packages/db/src/index.ts",
  "^src/(.*)$": "<rootDir>/$1"
}
Если ни одна секция не подошла — сначала посмотри логи процесса: 
pm2 logs escort-api --lines 100        # VPS
# или локально:
docker-compose -f docker-compose.dev.yml logs --tail=100 postgres
и .env.example как эталон переменных. Полный регламент по деплою — в ENTITY.md §6.
