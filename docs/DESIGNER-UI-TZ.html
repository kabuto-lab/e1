ТЗ для UI/UX-дизайнера

Lovnge — премиальная платформа бронирования с крипто-эскроу (TON USDT)

Версия: 2.1 Дата: 2026-04-21 Фаза: Phase 1 ≈ 90%

1. Контекст

Вертикальный продукт: каталог верифицированных моделей, кабинеты клиента/модели, CMS для менеджера/админа, оплата через TON USDT эскроу. Позиционирование — информационный сервис, премиальный сегмент.

Бэкенд, БД (Drizzle + PostgreSQL) и большая часть фронта (Next.js) уже реализованы. Единой дизайн-системы как Figma-файла нет — фронт собран по месту из DESIGN.md. Задача: собрать целостную систему, достроить недостающие экраны, привести существующие к единому стилю.

Продукт чувствителен к доверию. Визуал отвечает за восприятие безопасности и конфиденциальности сделки — не только за эстетику.

3. Дизайн-система locked-in

Все токены зафиксированы в DESIGN.md. Отклоняться без согласования нельзя — фронт уже собран на них.

  • Шрифты: два Google Fonts — display (заголовки, имена, логотип) и body (весь UI, формы, таблицы). Названия и веса — в DESIGN.md.
  • Палитра: dark-only, чёрный фон, серые поверхности, золотой акцент, статусные цвета success/warning/error. Все hex — в DESIGN.md.
  • Сетка: 12 колонок, max-width 1200px, базовый spacing 8px, радиус карточек 6–8px.
  • Моушен: 200–500ms, cubic-bezier(.4,0,.2,1), обязателен prefers-reduced-motion.
  • Фактура: лёгкий film grain 2–4% как декоративный слой, не поверх контента.
  • Направление: Luxury / Modern Luxury — сдержанное золото на чёрном, строгая сетка. Ассоциация: ювелирный бутик, не ночной клуб.

5. Зона ответственности дизайнера

  • User flows для трёх поверхностей: публичный сайт, кабинет клиента, дашборд.
  • Ре-дизайн существующих экранов + дизайн недостающих (список в §4).
  • Компонентная библиотека в Figma: variants/states, именование под фронт.
  • Адаптив: mobile-first от 320px → tablet → desktop 1200px.
  • Handoff-готовые макеты: именование слоёв, auto-layout, Figma variables.
  • Участие в планировании спринта: что сдать первым и почему.

6. Текущий спринт (нед. 14–15, апр 20–26)

Приоритеты, не блокирующие разработку:

  1. Media visibility UI — /dashboard/models/[id]/photos: тумблер is_public_visible, дропдаун album_category (portfolio / verification / private), bulk-действия. API готов.
  2. Экран "бронь + next action" — /cabinet/bookings: статус-бейдж, таймлайн 4 состояний эскроу, CTA под каждое состояние.
  3. BookingTonModal (полировка) — 4 шага: confirm → создание → инструкции+QR → funded+reveal. Нужно: размер QR, читаемость адреса, состояния loading/success/error, анимация funded.
  4. Карточка модели /models/[slug] — галерея + lightbox, параметры, отзывы, sticky-CTA.
  5. Каталог /models — фильтры, сетка карточек, все три состояния (загрузка / пусто / ошибка).

Не в этом спринте: Telegram-уведомления, фиат-оплата, закрытый клуб.

7. Ключевые сценарии

Для каждого flow — точка входа, happy path, alt-ветки, все состояния (loading / empty / error / success), конечное действие.

7.1. Бронирование через TON-эскроу

  1. /models/[slug] → "Забронировать". Не авторизован — редирект на /login, возврат в модалку.
  2. Confirm: модель, дата, длительность, сумма, комиссия, итог. CTA "Оплатить эскроу".
  3. Создание: memo, адрес кошелька, сумма USDT, QR-код.
  4. Инструкции: "отправьте X USDT с memo=Y на адрес Z". Live-поллинг.
  5. Funded: контакты раскрываются, видна кнопка "Подтвердить встречу" / "Открыть спор".

Статусы: draft → pending_payment → escrow_funded → confirmed → completed / disputed. Каждый — своя CTA и цвет бейджа.

7.2. Верификация модели

Статусы: pending → video_required → document_required → verified / rejected.

  • Модель видит текущий шаг и что загрузить.
  • Менеджер в /dashboard/moderation: очередь, карточка, approve/reject с комментарием.
  • Бейдж "verified" виден в каталоге и на карточке.

7.3. Медиа: видимость и альбомы

  • Два независимых флага: is_public_visible и album_category (portfolio / verification / private).
  • Grid миниатюр: toggle видимости + chip категории на каждой. Bulk-выделение и массовые действия.
  • На публичном профиле — только is_public_visible = true.

7.4. Модерация

Единая очередь (профиль / медиа / отзывы) с фильтрами. Карточка: diff "было/стало", комментарий, кнопки approve / reject / request-changes.

7.5. Блэклист

Таблица заблокированных с причиной. Фильтр, поиск, unblock с подтверждением.

9. Технические ограничения

  • Стек: Next.js 15 (App Router) + Tailwind CSS. Отступы — только по Tailwind-шкале, без произвольных значений.
  • Только тёмная тема — светлой нет и не будет.
  • Mobile-first, от 320px — каждый экран работает на маленьком экране.
  • Accessibility: контраст ≥ WCAG AA, prefers-reduced-motion, aria-labels для иконочных кнопок, фокус-кольца.
  • Без тяжёлых эффектов на публичных маршрутах (3D, шейдеры). Галереи — превью + ленивая загрузка.
  • Язык — русский. Закладывать длину строк как в русском (+15–20% к EN).
  • Безопасность UX: фильтрация телефонов/мессенджеров в переписке, визуальный feedback на rate-limit.

10. Handoff

Макет считается сданным, если разработчик открывает и собирает экран без дополнительных вопросов.

  • Осмысленные имена слоёв — никаких Group 341.
  • Компоненты с variants/states вместо дублей.
  • Auto-layout везде. Spacing, сетка, alignment зафиксированы.
  • Состояния совпадают с кодом: default / hover / focus / active / disabled / loading / error.
  • Figma variables вместо хардкода токенов.
  • Короткие аннотации к нетривиальной логике (напр., таймер холд-периода эскроу).
Не привязывать решения к WordPress-like допущениям — продукт на собственном стеке (Next.js + NestJS API).

11. Процесс

Дизайнер идёт на один шаг впереди разработки. Оценка — по качеству системы и скорости принятия решений, не по количеству страниц.

  1. Дизайнер прорабатывает flow + экраны + компоненты текущего спринта.
  2. Короткое согласование логики и визуала с владельцем продукта.
  3. Разработка реализует утверждённый блок; дизайнер доступен на вопросы.
  4. Пока разработка реализует — дизайнер готовит следующий блок.

Источник правды по статусу эпиков — вкладка "Статус" в apps/web/public/platform-blueprint.html.
