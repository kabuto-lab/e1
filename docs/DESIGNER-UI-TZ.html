Техническое задание для UI/UX-дизайнера

Проект: Lovnge — премиальная платформа бронирования эскорт-моделей с крипто-эскроу (TON USDT). Дизайнер проектирует пользовательские сценарии, достраивает существующую визуальную систему и готовит макеты спринт за спринтом под уже работающий фронтенд (Next.js).

Версия: 2.0 Дата: 2026-04-20 Фаза продукта: Phase 1 (Core CMS) ≈ 90% Формат: sprint-by-sprint, дизайнер идет на шаг впереди разработки

1. Контекст продукта

Что это. Вертикальный цифровой продукт — каталог верифицированных моделей, кабинеты для клиентов и моделей, CMS-дашборд для менеджеров/админов и защищенная оплата через TON USDT эскроу. Позиционирование — информационный сервис (не знакомства), оформленный под премиальный сегмент.

Стадия. Бэкенд, база (Drizzle + PostgreSQL) и большая часть фронта уже реализованы. Макетов как системы — нет: фронт сейчас собран "по месту" из DESIGN.md и платформенного блюпринта. Задача дизайнера — собрать это в целостную систему, достроить недостающие экраны и привести существующие к единому стилю.

Цель привлечения дизайнера. Снять блок "красивые макеты ≠ готовое состояние" и довести визуал и UX до продуктового уровня: чтобы клиент, модель и менеджер понимали свои действия на каждом шаге, а разработка получала макеты, готовые к реализации без пересборки.

Продукт чувствителен к доверию и статусу. Визуал — не украшение, а часть функции: он отвечает за восприятие безопасности, верификации и конфиденциальности сделки.

2. Роли и аудитория

Четыре роли с разными правами и разными интерфейсами. Дизайн обязан различать их в навигации, плотности информации и тоне.

Гость / Клиент

Просматривает каталог, добавляет в избранное, открывает бронирование, оплачивает эскроу, ведет историю встреч.

Модель

Ведет анкету, медиа-библиотеку, настройки видимости, видит брони, проходит верификацию, видит заработки.

Менеджер

Работает с несколькими моделями, обрабатывает заявки, комиссии, CRM-очередь, модерация контента.

Админ

Полный доступ: настройки платформы, пользователи, финансы, ЧС/блэклист, брендинг.

Кабинет клиента и дашборд менеджера/админа — это два разных продукта по навигации и плотности: клиент живет в нижней навигации и карточках, менеджер — в таблицах, очередях и модалках.

3. Существующая дизайн-система (locked-in)

Все токены и решения ниже зафиксированы в DESIGN.md. Отклоняться от них без согласования нельзя — продукт уже собран на них.

Шрифты

  • Unbounded (Google Fonts, 400–900) — заголовки, имена моделей, логотип, декоративные акценты.
  • Inter (Google Fonts, 300–700) — весь интерфейс, текст, таблицы, формы.

Палитра — dark-only (светлой темы нет и не планируется)

#0A0A0A — page bg
#1A1A1A — surface
#242424 — elevated / hover
#FFFFFF — text primary
#A0A0A0 — text secondary
#666666 — text muted
#D4AF37 — accent (gold)
#F4D03F — accent hover
#B8941F — accent pressed
#22C55E — success
#EAB308 — warning
#EF4444 — error

Сетка, отступы, ритм

  • Сетка: 12 колонок, max-width 1200px для публичной части и дашборда.
  • Базовый spacing: 8px. Все отступы кратны 4/8.
  • Карточки: padding 24px, radius 6–8px.
  • Вертикальный ритм между секциями: 48–64px.

Моушен и фактура

  • Анимации 200–500ms, easing cubic-bezier(.4,0,.2,1), обязательно уважение prefers-reduced-motion.
  • Легкая зернистая текстура (film grain) 2–4% — только как декоративный слой, не поверх контента.
  • Никаких тяжелых 3D/Three.js эффектов в продуктовом apps/web. Эксперименты (water shader и т.п.) живут отдельно в NON_PROJECT/.
Направление: Luxury / Modern Luxury — сдержанное золото на черном, строгая сетка, никаких "розовых градиентов" и дешевых эффектов. Ассоциация: ювелирный бутик, не ночной клуб.

5. Зона ответственности дизайнера

  • Проектирование user flows по трем поверхностям (публичный сайт, кабинет клиента, дашборд).
  • Достраивание единой визуальной системы на базе существующих токенов (не изобретать заново).
  • Ре-дизайн существующих экранов + дизайн недостающих по списку §4.
  • Адаптив: mobile-first (от 320px), tablet, desktop до 1200px.
  • Компонентная библиотека в Figma: variants/states, именование под фронт.
  • Handoff для разработки: понятная структура слоев, отступы, сетка, состояния.
  • Участие в планировании спринта: какие экраны дизайнер сдает в первую очередь и почему.

6. Приоритеты ближайшего спринта (нед. 14–15, апр 20–26)

Это не полный список задач, а то, что дизайнер должен закрыть в первую очередь, чтобы не блокировать разработку.

  1. Media visibility UI (эпик 3b) — страница /dashboard/models/[id]/photos: тумблер is_public_visible на миниатюре, дропдаун album_category (portfolio / verification / private), bulk-действия (выделить все, скрыть/показать). API готов, нужен только UI.
  2. Единый экран "бронь + next action" (5.17) — карточка брони в /cabinet/bookings: статус-бейдж, таймлайн из 4 состояний эскроу, одна CTA-кнопка с понятным текстом под каждое состояние.
  3. Доработка BookingTonModal (4-шаговый TON-флоу) — confirm → создание брони+эскроу → инструкции+QR → поллинг → reveal контактов. Сейчас закодировано, нужна визуальная полировка: размер QR, читаемость адреса кошелька, состояния loading/success/error, анимация funded.
  4. Публичная карточка модели /models/[slug] — галерея с lightbox, зона параметров, блок отзывов, sticky-CTA на бронирование.
  5. Каталог /models — фильтры (возраст, рост, параметры, верификация, доступность), сетка карточек, пустое/загруженное/ошибочное состояние.

Не в этом спринте: Telegram-уведомления, фиат-оплата (YooKassa), закрытый клуб, гостевая форма брони. Их можно держать в бэклоге дизайна.

7. Ключевые пользовательские сценарии

Для каждого сценария фиксируются: точка входа, шаги, все состояния (loading / error / empty / success), целевое действие.

7.1. Бронирование через TON-эскроу (ядро продукта)

  1. Клиент открывает /models/[slug], нажимает "Забронировать".
  2. Если не авторизован — редирект на /login, после входа возврат на модальное окно.
  3. Шаг 1 — Confirm: модель, дата, длительность, сумма, комиссия, итог. CTA "Оплатить эскроу".
  4. Шаг 2 — Создание: система выдает уникальный memo, адрес кошелька, сумму в USDT, QR-код.
  5. Шаг 3 — Инструкции: "отправьте X USDT с memo=Y на адрес Z". Live-поллинг индексатора.
  6. Шаг 4 — Funded: контакты модели раскрываются, запускается холд-период, видна кнопка "Подтвердить встречу"/"Открыть спор".

Статусы брони: draft → pending_payment → escrow_funded → confirmed → completed / disputed. Каждый статус — своя CTA и свой цвет бейджа.

7.2. Верификация модели

Статусы: pending → video_required → document_required → verified / rejected.

  • Модель в кабинете видит текущий шаг и что нужно загрузить.
  • Менеджер в /dashboard/moderation видит очередь, открывает карточку, approve / reject с комментарием.
  • Бейдж "verified" — видимый статус доверия в каталоге и на карточке.

7.3. Медиа: видимость и альбомы

  • Каждое изображение имеет два независимых флага: is_public_visible и album_category (portfolio / verification / private).
  • UI: grid миниатюр, на каждой — toggle видимости и chip с категорией. Bulk-выделение, массовые действия.
  • На публичном /models/[slug] отображаются только is_public_visible = true.

7.4. Модерация

Очередь: профиль / медиа / отзывы в одном списке с фильтрами. В карточке — diff "было / стало", комментарий модератора, кнопки approve / reject / request-changes.

7.5. ЧС / блэклист

Заблокированные пользователи с причиной (фейковые фото, no-show, мошенничество). Работа с таблицей: фильтр по причине, поиск, unblock с подтверждением.

Для каждого flow обязательно: стартовая точка, happy path, альтернативы, состояния загрузки/ошибки/пусто/успеха, конечное действие пользователя.

8. UI-kit: что нужно достроить

Токены (цвет/шрифт/сетка) уже зафиксированы. От дизайнера нужен системный компонентный слой.

Формы и инпуты

  • Text / textarea / select / multiselect / date-time picker / price-input (с валютой)
  • Все состояния: default, focus, filled, error, disabled, loading
  • Inline-валидация с понятным сообщением

Кнопки и CTA

  • Primary (золото), Secondary (outline), Ghost, Destructive
  • Размеры: sm / md / lg, иконка слева/справа, loading-стейт
  • Полноширинная кнопка для mobile CTA

Навигация

  • Публичный хедер + футер
  • Сайдбар дашборда (свернутый/раскрытый)
  • Нижняя навигация для mobile-кабинета

Карточки и списки

  • Карточка модели (каталог) + расширенная (профиль)
  • Карточка брони с timeline-статусом
  • Ряд таблицы дашборда, строка очереди модерации

Модалки и оверлеи

  • BookingTonModal — многошаговый, с индикатором шагов
  • Confirm / alert / delete
  • Lightbox галереи с keyboard-навигацией

Обратная связь

  • Toasts: success / error / info / warning
  • Empty states (с CTA и иллюстрацией)
  • Loading skeletons для списков и карточек
  • Страницы 401 / 403 / 404 / 500

9. Технические ограничения

  • Стек фронта: Next.js 15 (App Router) + Tailwind CSS. Макеты должны ложиться на Tailwind-утилиты — не рисовать произвольные значения вне спейсинг-шкалы.
  • Только темная тема. Светлой темы нет и не будет на этом этапе.
  • Mobile-first, минимум 320px. Все экраны обязаны работать на маленьком телефоне.
  • Accessibility: контраст не ниже WCAG AA, поддержка prefers-reduced-motion, aria-labels для иконочных кнопок, фокус-кольца.
  • Производительность: без тяжелых 3D/шейдеров на публичных маршрутах. Галереи — оптимизированные превью + ленивая загрузка.
  • Язык: только русский на этом этапе. Закладывать длину строк как в русском (на 15–20% длиннее EN).
  • SEO: модели имеют slug; для карточек продумать plug для OG-превью.
  • Безопасность UX: защита от утечек контактов (фильтрация телефонов/мессенджеров в сообщениях), rate-limit на формы (визуальная обратная связь).

10. Handoff для разработки

Макет считается сданным, если разработчик открывает его и собирает экран без дополнительных расшифровок.

  • Структура слоев и групп с осмысленными названиями (никаких Group 341).
  • Компонентный подход: variants/states вместо копий.
  • Фиксация spacing, сетки, размеров, align-правил. Auto-layout обязателен.
  • Именование состояний компонентов совпадает с тем, что будет в коде (default / hover / focus / active / disabled / loading / error).
  • Токены цветов и шрифтов — как Figma variables, не как хардкод.
  • Короткие заметки к нетривиальным зонам (например, логика таймера холд-периода эскроу).
Не привязывать решения к конкретной CMS/движку — продукт собирается на своем коде (Next.js + собственный NestJS API), никаких WordPress-like допущений.

11. Процесс работы (спринтами)

Дизайнер идет на один шаг впереди разработки. Оценка — не по количеству отрисованных страниц, а по качеству системы и скорости принятия решений.

  1. Шаг 1: дизайнер прорабатывает flow + экраны + нужные компоненты текущего спринта.
  2. Шаг 2: короткое согласование логики и визуала с владельцем продукта.
  3. Шаг 3: разработка реализует утвержденный набор (Claude Code + инженер), дизайнер доступен на вопросы.
  4. Шаг 4: дизайнер уже готовит следующий блок, не дожидаясь завершения всех страниц.

Важно Источник правды по статусу эпиков — вкладка "План → Статус" в apps/web/public/platform-blueprint.html. Дизайнер сверяется с ней перед стартом спринта.

13. Формат передачи и рабочие принципы

  • Обязательно Один источник правды — актуальный Figma-файл с библиотекой компонентов и экранами.
  • Обязательно Все изменения токенов проходят согласование и отражаются в DESIGN.md.
  • Ограничение Не привязывать решения к конкретному движку/CMS.
  • Ограничение Не использовать стоковые иллюстрации низкого качества и клишированные эскорт-визуалы — продукт премиальный.
  • Принцип Сначала логика и система, потом масштабирование количества страниц.
  • Принцип Каждое UI-решение должно работать на доверие и ощущение безопасности сделки.

Итог: дизайнер поставляет продуманную систему и последовательность шагов, после утверждения разработка реализует блок за блоком без хаотичных переработок.
