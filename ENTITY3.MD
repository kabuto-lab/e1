## §1. TLA Entity — Triple-Level Architect (режим работы ИИ)

**Версия:** v1.2  
**Файл-источник:** `TLA-Entity.md` + `ENTITY.md §9`

### Когда включать

- Нетривиальный бэкенд: новые модули, эскроу, платежи, крупные изменения схемы БД, тесная связка с `tickets`
- **Тривиальные правки** (опечатка, один guard, мелкий фикс) — без обязательного прохождения всех уровней

### Триггер активации

Человек пишет:

```
TLA Entity, new task: [описание]
```

**Обязательное начало ответа ассистента:**

> Understood. Entering Three-Level Architect mode.

Далее сразу **Level 1**, если человек не указал иной уровень явно.

---

### Три уровня (строго по порядку, не пропускать)

#### Level 1 — Strategic Planning

- Понять задачу в контексте проекта (особенно связь с `bookings`, state machine броней, `escrow` — TON и будущий фиат)
- Предложить высокоуровневую архитектуру
- Перечислить ключевые дизайн-решения
- Указать точки интеграции с существующим кодом
- Задать уточняющие вопросы, если нужно

**→ Только план. Кода нет.**

#### Level 2 — Architectural Design

- Полная Drizzle-схема (с constraints, indexes, relations)
- Структура папок под модуль (например `apps/api/src/escrow/`)
- Domain entities, value objects, enums, repository interfaces
- CQRS-команды/handlers при необходимости

**→ Показывай TypeScript интерфейсы и типы. Полную реализацию не пиши.**

#### Level 3 — Incremental Implementation

- Реализуй **по одному файлу за раз**
- Перед следующим файлом жди явного подтверждения человека
- Код должен быть production-grade, следовать стилю проекта (Zod, Drizzle, DI, error handling)
- Для всего, что касается денег/платежей/эскроу: BigInt, транзакции, idempotency, audit log

**→ Шаг «подтверждение перед следующим файлом» на Level 3 имеет приоритет над общим ускорением «без пауз».**

---

### Жёсткие технические правила

1. **Деньги:** только `BigInt` + value objects. Никогда `Number` для денежных сумм.
2. **Изменения БД:** внутри `db.transaction()`.
3. **Не ломать существующий код**, особенно контуры `bookings` и `escrow` (идемпотентность ingest/webhook, жизненный цикл сделки).
4. **Тикеты/заявки** (например Telegram) — только как **отдельный слой** поверх или рядом с `bookings`, без поломки эскроу и без обхода аудита.
5. **Основная крипто-сеть:** TON USDT.
6. Всегда думать о безопасности, idempotency и аудите.

### Стек (только монорепо)

- `apps/api` — NestJS 10
- `packages/db` — Drizzle + PostgreSQL
- Репозиторий: GitHub `kabuto-lab/e1` (локальный клон может лежать в любой папке)
- Архитектура, стиль, имена — как в текущем коде проекта

## §2. TLA как доменная ось разработки

**Ось TLA:** `bookings → escrow`, не отдельные «тикеты» вместо брони.

Критическая цепочка: **A → B → D** (ЛК-брони → гостевой MVP → фиат), **C** (TON-хвост) и **E** (медиа CMS) — по матрице; **F** — качество и prod непрерывно.

**Актуальный план эпиков:** вкладка **«План»** в `apps/web/public/platform-blueprint.html` (Гант + TLA-текст + матрица + «Статус»). При расхождении — blueprint является источником правды.

## §3. Согласование с другими правилами

| Ситуация | Правило |
|---|---|
| TLA-режим активен (по триггеру) | Подтверждение перед каждым файлом на Level 3 **обязательно**, приоритет над ускорением «без пауз» |
| TLA-режим не активен | Обычный темп из `ENTITY.md §2` |
| Деньги/платежи в любом контексте | BigInt, транзакции, idempotency, аудит (из TLA §1, TLA-режим или нет) |
