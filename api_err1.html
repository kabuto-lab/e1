API Incident Report: PostgreSQL Authentication Mismatch

File: api_err1.html
Date: 2026-04-07
Environment: VPS (Linux), API managed by PM2, PostgreSQL in Docker container escort-postgres

1) Executive Summary

The API returned database authentication errors during login and model catalog loading. The root issue was a mismatch between credentials in DATABASE_URL and the actual password configured for PostgreSQL role postgres in the running database instance. Even though .env looked correct, the real DB role password did not match it.

2) User-Facing Symptoms

  • Login failed with an API error about PostgreSQL rejecting connection credentials.
  • Model catalog page failed with: "Could not load catalog".
  • Both issues were secondary effects of API failing to authenticate to PostgreSQL.

3) Initial Hypotheses

  • Hypothesis A: DATABASE_URL in API process was wrong.
  • Hypothesis B: PostgreSQL service was down/unreachable on host:port.
  • Hypothesis C: API process had stale environment variables after deployment/restart.
  • Hypothesis D: Real DB role password had drifted from values stored in .env.

4) Diagnostic Steps Performed

4.1 Validate current DATABASE_URL read from .env

cd ~/e1
node -e "require('dotenv').config(); console.log(process.env.DATABASE_URL)"

Observed value:

postgresql://postgres:postgres@127.0.0.1:5432/companion_db

4.2 Validate network binding and container mapping for PostgreSQL

ss -tlnp | grep 5432
docker ps --format 'table {{.Names}}\t{{.Ports}}' | grep -E 'escort-postgres|5432'

Result confirmed that host 127.0.0.1:5432 was mapped to container escort-postgres.

4.3 Test DB auth exactly as API would (TCP from host)

PGPASSWORD=postgres psql -h 127.0.0.1 -p 5432 -U postgres -d companion_db -c "select 1;"

Result:

FATAL:  password authentication failed for user "postgres"

This was the decisive signal: endpoint/host/port was correct, but credentials were not accepted by PostgreSQL.

5) Root Cause

Credential drift between application config and live database role password. Specifically, DATABASE_URL expected postgres:postgres, while role postgres in the running PostgreSQL instance had a different password at runtime.

In Dockerized PostgreSQL setups, changing POSTGRES_PASSWORD in compose files does not always update password inside an already initialized database volume. This commonly creates hidden mismatches.

6) Fix Applied

Used the project maintenance script to align DB role password with DATABASE_URL:

cd ~/e1
npm run db:align-password

Output:

ALTER ROLE
OK: password for role postgres in container matches DATABASE_URL (.env)

Then revalidated DB and reloaded API process with fresh environment:

cd ~/e1
npm run ensure:database
npm run pm2:reload-api

7) Why Both Errors Happened Together

  • Login failure: API could not query auth/user tables because DB auth failed.
  • Catalog failure: API could not query models data; frontend surfaced generic catalog load error.
  • Single root cause: one DB auth mismatch produced multiple endpoint failures.

8) Important Operational Lessons

  • Always verify credentials by an actual TCP login test from host, not only by reading .env.
  • Prefer npm run vps:after-pull after deployment to ensure build + DB check + proper API reload.
  • Avoid plain pm2 restart escort-api when env changes are involved; use project reload flow.
  • Automated guardrails should fail fast before API accepts traffic with broken DB credentials.

9) Preventive Checks to Add Permanently

  • Pre-start DB connectivity check in API startup pipeline.
  • Credential drift detector that compares env user/password to live DB auth handshake.
  • Deployment health gate that blocks restart success if DB auth test fails.
  • Clear runbook with one-command recovery path and expected outputs.
