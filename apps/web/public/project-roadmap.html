1. Краткое резюме

Платформа: CMS профилей моделей, публичный каталог, медиа (MinIO), задел под бронирования, эскроу, отзывы, CRM-клиентов, чёрные списки. Инфраструктура и базовая безопасность API (CORS, Helmet, валидация, Swagger) в значительной степени на месте. Часть старых аудитов устарела: видимость медиа и DESIGN.md уже есть в коде/репо.

Критические направления: убрать дыры в JwtAuthGuard на отдельных маршрутах, замкнуть дашборд на реальные API, ввести тесты, завершить публичный премиум-UX (слайдер, шейдер), затем бронирования и платежи.

2. Архитектура (факт)

Слой Содержание
apps/api NestJS: auth, users, models, profiles, clients, bookings, escrow, reviews, blacklist, media, health, database + guards, rate limit
apps/web Next.js App Router: лендинг, логин, публичные модели, дашборд (модели, медиа, фото, home, настройки), моки бронирований/модерации
packages/db Drizzle-схемы: пользователи, профили моделей/клиентов, медиа (в т.ч. видимость, альбомы), брони, эскроу, отзывы, аудит и др.

Внимание Параллельно существуют модули models и profiles — нужна явная граница ответственности и публичный контракт.

3. Сильные стороны

  • Безопасность базового уровня: env, Helmet, CORS с белым списком, ValidationPipe, Swagger.
  • RBAC: роли admin, manager, model, client; RolesGuard на части эндпоинтов.
  • Медиа: поля видимости и альбомов в БД; UI медиатеки и фото модели с переключением видимости.
  • Дизайн-система: DESIGN.md, шрифты Unbounded + Inter на вебе.
  • Публичные страницы: каталог и профиль по slug с учётом isPublicVisible.

4. Риски и долги

  • API В models.controller часть маршрутов с отключённым JWT — риск для прода.
  • Тесты Практически нет автотестов приложений — регрессии не ловятся в CI.
  • UI Бронирования и модерация в дашборде на моках; «Клиенты» и «Финансы» — заглушки (#).
  • UX Навигация дашборда не фильтруется по роли (в отличие от реальных прав API).
  • Визуал Fade slider и water shader из плана в продуктовом коде не подключены.
  • Документы CLAUDE.md и часть аудитов не отражают текущий прогресс (visibility, DESIGN).

5. Оценка готовности (ориентир)

Область Оценка Комментарий
Инфраструктура, Docker, Swagger 85–90% Сильная база
Auth + RBAC (реализация) ~75% Есть дыры в guard’ах на отдельных путях
CMS моделей + медиа 70–80% Значительный прогресс к старым аудитам
Публичный каталог / профиль 65–75% Нужны слайдер, шейдер, полировка
Бронирования end-to-end 25–35% UI в основном мок
Модерация в продукте 25–35% Мок + задел в схеме медиа
Платежи / эскроу в бою 15–25% Схема есть, провайдер не подключён
Тесты, CI 5–15% Требуется внедрение

Итоговая фаза: Phase 1 (Core CMS) ориентировочно 65–75%, а не зафиксированные ранее 60% по устаревшему чеклисту.

Связанные артефакты: по неделям · Гант (вкладка в blueprint) · platform-blueprint.html — матрица ролей и процессов; DESIGN.md — визуальные правила; Swagger — /api/docs на бэкенде.

6. Роадмэп по эпикам

Эпик A — Безопасность и контракт API 1–2 недели
  • Аудит контроллеров: включить JwtAuthGuard везде, кроме явно публичных маршрутов.
  • Зафиксировать публичный контракт для каталога и профиля; согласовать роль models vs profiles.
  • Минимальные integration/e2e-тесты: логин, 2–3 критичных сценария.
  • Актуализировать Swagger и описания ролей.
Эпик B — Дашборд по ролям и реальные данные 2–3 недели
  • Фильтрация пунктов меню по роли (согласовано с матрицей доступа).
  • Бронирования: замена моков на API, статусы из БД, действия менеджера.
  • Модерация: очередь по moderationStatus и связанным сущностям; approve/reject через API.
  • Клиенты и финансы: реализация страниц или скрытие до готовности.
Эпик C — Публичный опыт и премиум-визуал 2–4 недели
  • Компонент FadeSlider на публичном профиле (только видимые медиа, lazy load, мобильные).
  • WaterShaderOverlay или упрощённый вариант + fallback без WebGL.
  • Галерея / лайтбокс; Open Graph и мета для шаринга.
  • Приведение ключевых публичных экранов к DESIGN.md.
Эпик D — Бронирование и деньги 3–6 недель
  • Сквозной сценарий: запрос брони → статусы → подтверждение моделью/менеджером; строгая машина состояний на API.
  • Интеграция эскроу с платёжным провайдером (или поэтапно: ручной/полуавтоматический режим MVP).
  • Уведомления: от Mailhog в dev к продакшен-почте.
Эпик E — Качество и эксплуатация параллельно
  • CI: lint, typecheck, тесты; при готовности — деплой.
  • Миграции Drizzle и единообразие окружений.
  • Логи, корреляция с аудитом.
  • Обновить CLAUDE.md и COMPREHENSIVE_AUDIT_AND_PLAN.md под фактическое состояние.

Визуальная шкала (не календарь)

Период Фокус
Недели 1–3 A (безопасность API) + старт B (роли в меню) + старт E (CI smoke)
Недели 4–7 B (брони и модерация на API) + начало C (слайдер)
Недели 8–14 C (шейдер, полировка публички) + D (сквозная бронь и платежи MVP)

7. После MVP (отложено)

  • Интеграции мессенджеров (Telegram/WhatsApp) для CRM.
  • Расширенная аналитика, A/B, психотипы, VIP-воронки.
  • 2FA, сброс пароля, верификация e-mail — по требованиям комплаенса.
  • Альтернативные способы оплаты (отдельный эпик от MVP-эскроу).

8. Рекомендации архитектору

  1. Ввести один источник правды: короткий ARCHITECTURE.md или раздел в репо — границы модулей, публичный API, матрица ролей, статус эпиков.
  2. Платежи: сначала явный MVP-контур «бронь → статус → ручной/полуавтоматический эскроу», затем провайдер.
  3. Не допускать расхождения документации и кода — обновлять проценты и «следующие шаги» после каждого спринта.
