Тз разраб оплата
ТЗ: двухстадийная оплата  (hold / capture)
1. Цель
Реализовать оплату бронирования через холд (заморозку средств) с последующим списанием или отменой.
Без внутреннего кошелька и хранения денег на стороне платформы.
2. Сценарий
1. Клиент создаёт заявку
2. Нажимает «Оплатить»
3. Создаётся платёж с холдом (двухстадийная оплата)
4. Деньги замораживаются
5. После оказания услуги:
админ нажимает «Подтвердить» → списание (capture)
6. При отмене:
админ нажимает «Отменить» → снятие холда (cancel/void)
3. Интеграция
Подключить 1 из провайдеров:
1. CloudPayments
2. Юкасса
3. Тинькофф Касса
Требование:
использовать двухстадийную оплату (hold → capture)
без автосписания
4. Backend (обязательно)
Методы:
createPaymentHold(order_id, amount)
capturePayment(order_id)
cancelPayment(order_id)
getPaymentStatus(order_id)
processWebhook(payload)

Обязательно:

9. Бизнес-правила
capture только вручную
cancel только до оказания услуги
при истечении холда → статус expired
повторное списание запрещено
10. Cron
Каждые 5–10 минут:
проверка холдов
если истёк → expired
уведомление админа
11. UI (минимум)
После оплаты показать:
«Бронирование создано. Средства временно заблокированы и будут списаны после подтверждения услуги.»

14. Критерий готовности
клиент может оплатить → холд проходит
админ может сделать capture
админ может отменить
статусы синхронизируются
webhook работает
ошибок двойного списания нет

Платежная система
1. Модель (фиксируем основу)
Мы = агрегатор (не исполнитель)
Это значит:
Мы не оказываем услугу
Мы соединяем клиента и исполнителя
Мы берёшь комиссию
Это важно, иначе нас будет блокировать любая платёжка.
2. Как должна выглядеть оплата (рабочая схема)
Поток денег
1. Клиент оплачивает на сайте
2. Деньги НЕ остаются у нас
3. Платёжка держит их временно (холд)
4. После услуги:
% → нам
остальное → исполнителю
3. Конкретная реализация (самая стабильная)
Используем:
CloudPayments
или
Тинькофф Касса
Почему:
лучше проходят «серые» ниши
есть холд платежа
есть API для логики
4. Логика внутри системы (очень важно)
1. Создание заявки
Клиент:
выбирает девушку
выбирает услугу
нажимает «оплатить»
2. Платёж (HOLD)
Мы  делаем:
холд платежа (НЕ списание)
Что это значит:
деньги «заморожены» у клиента
но не списаны окончательно
3. Статусы (обязательно)
У нас должны быть статусы:
created — заявка создана
paid_hold — деньги заморожены
in_progress — услуга началась
done — услуга завершена
cancelled — отмена
4. После услуги
Есть 2 варианта:
Вариант А (простой)
Мы списываем деньги (capture)
Дальше:
переводим исполнителю вручную / через выплаты
Вариант Б (правильный, но сложнее)
сразу делим платёж:
% → исполнителю
% → нам
(через marketplace-логику)
5. Как НЕ получить блокировку (очень важно)
1. Формулировка бизнеса
НЕ пишем:
«услуги сопровождения»
«интим»
«эскорт»
Пишим:
«сервис подбора досуга»
«консьерж-сервис»
«организация встреч»
2. Оферта
Обязательно:
Мы пишем:
«платформа не является исполнителем»
«платформа предоставляет доступ к анкете исполнителей»
«оплата — сервисный сбор + бронь»
3. Деньги
Самое главное правило:
НЕЛЬЗЯ:
хранить деньги пользователей
делать «баланс внутри сайта»
МОЖНО:
холд
списание
комиссия
6. Минимальный MVP (как стартовать быстро)
Делаем так:
1. CloudPayments
2. Только HOLD
3. После услуги → вручную подтверждаем
4. Выплаты исполнителям вручную
7. Как это выглядит для клиента
Для него:
оплатил → деньги «списались»
услуга прошла
всё
Он не видит никакого escrow
