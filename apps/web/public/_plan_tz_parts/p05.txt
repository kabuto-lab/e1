Как настроить тг и вц
ЧАСТЬ 1. НАСТРОЙКА TELEGRAM
Шаг 1. Создать бота
1. В Telegram найти @BotFather
2. Нажать Start
3. Ввести команду:
/newbot
4. Придумать название
5. Придумать username (обязательно с bot в конце)
6. Получить токен (это ключ для разработчика)
Токен сразу сохранить.
Шаг 2. Подключить сервер
Дальше нужен разработчик.
Он должен:
поднять сервер (VPS или облако)
подключить Telegram Bot API
сделать обработку входящих сообщений
Принцип работы:
Клиент → бот → сервер → база данных → интерфейс менеджера → ответ → бот → клиент

Шаг 3. Создать роли
В системе должны быть роли:
Admin
Manager
Performer (девушка)
Разработчик делает проверку по Telegram ID.
То есть:
каждый сотрудник добавляется в систему по своему Telegram ID
бот понимает, кто он и что ему показывать

Шаг 4. Разграничение доступа
Менеджер:
видит только свои заявки
Девушка:
видит только свои заявки
Админ:
видит всё
Это настраивается через базу данных и проверки ролей.

Шаг 5. Логирование
В базе данных должны храниться:
ID заявки
Telegram ID клиента (скрыто)
История переписки
Кто отвечал
Время ответа
Статус
Это обязательно.

ЧАСТЬ 2. НАСТРОЙКА WHATSAPP
Тут важно сделать правильно.
❗ Нельзя использовать обычный телефон сотрудника.
Нужен WhatsApp Business API.

Шаг 1. Зарегистрировать WhatsApp Business
1. Создать Facebook Business Manager
2. Пройти верификацию компании
3. Подключить WhatsApp Business Account
4. Подключить номер (отдельный, не личный)

Шаг 2. Подключить провайдера API
Нужно выбрать официальный провайдер:
Примеры:
360dialog
WATI
Twilio
MessageBird
Они дают API-доступ.

Шаг 3. Подключение к серверу
Разработчик:
подключает API к вашему серверу
настраивает webhooks (уведомления о входящих)
сохраняет сообщения в базу
Схема такая же:
Клиент → WhatsApp → API → сервер → база → интерфейс → ответ → API → клиент

ЧАСТЬ 3. Как скрыть номер клиента
Это делается автоматически, если:
сотрудники работают НЕ в WhatsApp
сотрудники работают в вашей CRM/панели
То есть:
Они отвечают не в WhatsApp,
а в интерфейсе системы.
Система сама отправляет сообщение клиенту.
Номер хранится в базе в закрытом поле.

ЧАСТЬ 4. Интерфейс сотрудников
Есть 2 варианта.

Вариант А — Telegram как интерфейс (быстро)
Сотрудники работают через отдельного Telegram-бота.
Плюс:
быстро
дешево
не нужно делать веб-панель
Минус:
ограниченный интерфейс

Вариант B — Веб-панель (правильно)
Разработчик делает:
страницу входа
список заявок
чат-интерфейс
статусы
фильтры
Это дороже, но масштабируемо.

ЧАСТЬ 5. Минимальный стек
Backend:
Python (FastAPI) или Node.js
PostgreSQL
Интеграции:
Telegram Bot API
WhatsApp Business API
Хостинг:
любой VPS (DigitalOcean, Hetzner, REG.RU и т.д.)

ЧАСТЬ 6. Что обязательно предусмотреть
1. Фильтр отправки номеров в чатах
2. Запрет на пересылку контактов

ЧАСТЬ 7. Сроки
Telegram — 3–7 дней
WhatsApp API — 7–14 дней (зависит от верификации компании)
Полная система — 2–4 недели
Тз для разработчика вц и тг
ТЗ разработчику
Проект: Премиальная платформа заявок и чатов (Telegram + WhatsApp)
Цель: все заявки и переписки проходят через платформу; сотрудники (менеджеры/исполнители) не видят реальный номер клиента и не видят чужие переписки.
0) Термины и роли
Клиент — человек, который пишет в Telegram/WhatsApp.
Менеджер — сотрудник, который может брать/вести заявки.
Исполнитель — сотрудник (девушка), который может вести заявки, назначенные на неё.
Админ — владелец/управляющий, видит всё.

1) Каналы связи (обязательно)
1.1 Telegram (обязательно)
Клиент пишет в Telegram-бота.
Переписка сохраняется в системе.
Ответ сотрудника отправляется клиенту через бота.
1.2 WhatsApp (обязательно)
Клиент пишет на официальный WhatsApp Business номер (через WhatsApp Business API провайдера).
Все входящие/исходящие сообщения проходят через сервер проекта и сохраняются в системе.
Сотрудник НЕ общается с клиентом из личного WhatsApp.
Важно: WhatsApp реализуется через официальный WhatsApp Business API (провайдер выбирается отдельно: 360dialog/Twilio/MessageBird/WATI и т.п.).

2) Ключевые требования безопасности
1. Сотрудники не видят номер телефона клиента (в интерфейсе вместо номера показываем: Клиент #C-1047или Заявка #A-2041).
2. Сотрудники не видят чужие чаты:
Менеджер видит только свои заявки.
Исполнитель видит только свои заявки.
Админ видит всё.
3. Все сообщения логируются и доступны админу.
4. Нельзя “слить контакты”:
фильтрация попыток отправки телефона/контактов (минимум: предупреждение + логирование события; максимум: блокировка отправки).
отдельный лог “попытки обмена контактами”.

3) Основная логика (пользовательский сценарий)
3.1 Создание заявки
При первом сообщении клиента создаётся карточка заявки:
ID заявки
канал (TG/WA)
внутренний ID клиента
дата/время
текст первого сообщения
статус: Запрос
поле Ответственный = пусто
3.2 Назначение ответственного
Нужно 2 режима (переключаемые настройкой админом):
Режим 1: ручной — заявка падает в общий “входящие”, менеджер/админ/исполнитель может взять.
Режим 2: по карточке анкеты — если заявка пришла из конкретной анкеты (передаётся profile_id), то назначать сразу исполнителю (или менеджеру анкеты).
3.3 Переписка
Клиент пишет → сообщение появляется у ответственного в интерфейсе → ответственный отвечает → сообщение уходит клиенту в исходный канал.
Вся история чата хранится в системе.
3.4 Статусы
Обязательные статусы:
Запрос
В работе
Подтверждение
Встреча
Закрыта
Дополнительные:
Отказ
Не отвечает
ЧС

4) Интерфейс сотрудников (MVP)
Разработать веб-панель (рекомендуется) или Telegram-интерфейс.
Так как вы писали “без разницы”, в ТЗ фиксируем веб-панель как основной (иначе WhatsApp нормально не масштабируется).
4.1 Авторизация
Вариант MVP: вход по одноразовому коду/паролю для сотрудников + роль.
Привязка сотрудника к Telegram ID (для уведомлений).
4.2 Список заявок
Фильтры:
Мои заявки
Входящие (только для менеджеров/админов, если режим ручной)
Статус
Канал (TG/WA)
В строке заявки показывать:
ID
канал
время последнего сообщения
статус
ответственный
тег “новое”
4.3 Карточка заявки
Внутри:
чат
статус (смена статуса)
назначение/смена ответственного (доступ зависит от роли)
внутренние заметки (видит только персонал)
теги/метки (VIP/повтор/риск)
лог событий (взял в работу, сменил статус, попытка контакта и т.д.)

5) Уведомления
5.1 Уведомления сотрудникам
При новом сообщении в их заявке — уведомление в Telegram (личное сообщение сотруднику).
При назначении заявки — уведомление в Telegram.
5.2 Уведомления админу
Новая заявка
Долгое молчание по заявке (например 10 минут без ответа) — как событие/алерт (можно MVP: просто отметка/фильтр “без ответа > 10 минут”).

6) Контакты клиента (как храним и как показываем)
6.1 Хранение (в БД)
Telegram: telegram_user_id, username (если есть)
WhatsApp: phone_e164 (номер в E.164)
6.2 Отображение персоналу
Никогда не показываем telegram_user_id и phone_e164
Показываем только:
Клиент #C-XXXX
Заявка #A-XXXX

7) Интеграции
7.1 Telegram Bot API
Функции:
принимать входящие сообщения
отправлять ответы
поддержка текста + медиа (минимум фото; видео по возможности)
7.2 WhatsApp Business API (через провайдера)
Функции:
webhooks на входящие сообщения
отправка исходящих
поддержка текста + медиа (минимум фото; видео по возможности)

8) Анти-слив (контроль контактов)
MVP-реализация:
если сотрудник пытается отправить сообщение, похожее на телефон/контакт:
система показывает предупреждение “нельзя передавать контакты”
событие логируется в карточке заявки (время, кто, что пытался отправить)
сообщение не отправляется клиенту (или отправляется с маской — на ваш выбор; в ТЗ фиксируем: не отправляется)
Детали фильтра:
регулярка на телефоны РФ и общий E.164
слова-триггеры: “звони”, “тел”, “whatsapp”, “ватсап”, “tg”, “телеграм”, “@”, “instagram” (минимальный словарь)

9) Чёрные списки (MVP)
9.1 ЧС клиентов
сущность “blacklisted_clients”
если клиент в ЧС:
в заявке показывается бейдж
новые обращения помечаются “ЧС”
возможность блокировки обработки (по настройке)
9.2 ЧС исполнителей/профилей (внутренний)
Можно отдельным модулем позже. Для MVP достаточно ЧС клиентов + статусов заявок.

10) Данные и база
Рекомендуемый стек:
Backend: Python FastAPI / Node.js
DB: PostgreSQL
Очередь/фоновые задачи: optional (Celery/RQ/Bull) — для уведомлений
Хранилище медиа: S3-совместимое или локально (на MVP допускается локально)
10.1 Минимальные таблицы
users (персонал)
roles
clients
tickets (заявки)
messages (сообщения)
ticket_events (лог)
blacklisted_clients

11) Права доступа (ACL)
Жёстко:
Admin: видит все заявки/чаты/логи, может назначать/переназначать.
Manager: видит только свои + может брать из “входящих” (если включено).
Performer: видит только назначенные на него/неё заявки.
Запрет просмотра по URL чужих заявок (проверка на backend).

12) Логи и отчёты (MVP)
В админке минимум:
количество новых заявок за день
сколько в работе/закрыто
среднее время первого ответа (если успеете)
экспорт CSV заявок за период (минимум: 24 часа)

13) Нефункциональные требования
Хранение сообщений и статусов без потерь
Бэкапы БД (минимум ежедневные)
Логи ошибок + алерт админу в Telegram при критике (упал webhook, недоступна БД и т.п.)
Таймзона: Europe/Moscow или Europe/Berlin (зафиксировать в конфиге; по умолчанию Europe/Moscow)

14) Этапы разработки и критерии приёмки
Этап 1 — Telegram (MVP)
Готово, если:
клиент пишет боту → создаётся заявка
менеджер/исполнитель отвечает из панели → клиент получает ответ
сотрудник видит только свои заявки
админ видит все
Этап 2 — WhatsApp (MVP)
Готово, если:
WA сообщение создаёт заявку
ответ из панели уходит клиенту в WA
номер клиента нигде не отображается персоналу
Этап 3 — Роли/ACL/Логи/Анти-слив
Готово, если:
права доступа работают
события логируются
фильтр контактов работает и пишет событие

15) Что предоставить в конце (обязательно)
репозиторий с кодом
файл .env.example + инструкция развертывания
схема БД (миграции)
инструкция для админа: как добавлять сотрудников, роли, как смотреть заявки
список всех webhook URL и параметров для WhatsApp провайдера
чеклист тестирования (короткий)
