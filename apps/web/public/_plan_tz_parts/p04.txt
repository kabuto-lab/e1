Для пм
Проект: Премиальная платформа сопровождения
Документ для Project Manager
Общая задача
Нам нужно запустить премиальную онлайн-платформу подбора и бронирования проверенных компаньонов для мероприятий и частных встреч.
Ключевая идея проекта — доверие и безопасность.
Не массовый каталог, а закрытая система с контролем качества.
Твоя задача как Project Manager — довести проект до полноценного запуска и первых оборотов. Ты отвечаешь за результат целиком.
Что мы строим
Это не просто сайт с анкетами.
Это система, в которой:
анкеты проходят проверку
есть статус и прозрачная история
используется безопасная сделка
фиксируются все заявки
есть внутренняя система контроля
Проект должен выглядеть дорого, аккуратно и спокойно. Без провокаций. Это премиальный сегмент.
Основная логика продукта
1. Каталог
Фильтры:
возраст
рост
цвет волос
знание языков
параметры
статус VIP / Elite
психотип
Отдельно будет подбор по психотипу: клиент проходит короткий тест и получает персональную подборку.
2. Карточка анкеты
Каждая анкета должна включать:
10–15 живых фото
видео
короткое видео с текущей датой
дату последнего подтверждения
отзывы
статус «Проверена»
live-статус (в смене / свободна после 20:00 и т.д.)
блок «Проверка» (документы подтверждены, фото проверены и т.п.)
Нам важно, чтобы карточка выглядела как профиль в закрытом клубе, а не как обычное объявление.
3. Статус Elite
Чтобы получить Elite, анкета должна:
пройти личное интервью
пройти видео-аудит
не иметь негативных отзывов
Это должно быть реальное отличие, а не просто бейджик.
Безопасная сделка
Работаем через модель эскроу.
Схема простая:
1. Клиент оплачивает через платформу
2. Деньги замораживаются
3. После подтверждения встречи средства перечисляются исполнителю
4. Комиссия удерживается автоматически
Project Manager должен:
подобрать платёжное решение
согласовать модель с юристом
предусмотреть резервный вариант на случай блокировок
Комиссионная модель
Менеджеры: 10–20% (на старте 3%)
Прямые анкеты: 25–30% (на старте 3%)
Возможно — подписка за размещение
Финансовая модель должна быть просчитана до запуска.
CRM и контроль
Все заявки идут только через платформу.
Нужно реализовать:
фиксацию клиента
журнал действий
статусы сделки:
Запрос

Встреча
Закрыта

доступ к VIP
бонусную систему

минимализм

без пошлости

Общение в тг и вц
1. Общая логика системы
Принцип простой:
Клиент общается только с сервисом
Менеджеры и девушки работают внутри системы
Никто из них не видит номер клиента
Никто из них не видит чужие переписки
Всё логируется и хранится в карточке заявки
2. Точки входа клиента
1️⃣ Telegram
Клиент пишет в Telegram-бота платформы.
2️⃣ WhatsApp
Клиент пишет на официальный WhatsApp Business номер платформы.
Оба канала подключаются к одной CRM-системе.
3. Что происходит технически
Шаг 1. Создание заявки
При первом сообщении автоматически создаётся:
Карточка заявки:
ID заявки
источник (TG / WA)
ID клиента (внутренний)
время
текст первого сообщения
статус: Запрос
поле “ответственный” (пока пусто)
Номер телефона и Telegram ID не показываются сотрудникам.
В системе они хранятся в скрытом виде.
4. Назначение ответственного
Заявка попадает в общий “пул”.
Дальше возможны 2 сценария:
Сценарий A — менеджер
Менеджер берёт заявку → становится ответственным.
Сценарий B — конкретная девушка
Если клиент писал с конкретной карточки анкеты — заявка сразу назначается этой девушке.
5. Переписка
Клиент ↔ Система ↔ Ответственный
Telegram:
Клиент пишет боту → сообщение появляется в интерфейсе ответственного → ответственный отвечает → бот отправляет клиенту.
WhatsApp:
Клиент пишет в WA → сообщение попадает в CRM → ответственный отвечает → сообщение уходит клиенту от имени бизнес-аккаунта.
Важно:
Девушка НЕ видит номер клиента
Менеджер НЕ видит номер клиента
Клиент НЕ видит личные контакты сотрудников
6. Разграничение доступа (очень важно)
Менеджер:
видит только заявки, которые он взял
не видит переписки других менеджеров
не видит переписки девушек
Девушка:
видит только свои заявки
не видит заявки других девушек
не видит заявки менеджеров
Администратор:
видит всё
Это реализуется через систему ролей:
Role: Admin
Role: Manager
Role: Performer (девушка)
7. Статусы заявки
Минимальный пайплайн:
1. Запрос
2. В работе
3. Подтверждение
4. Встреча
5. Закрыта
Дополнительные:
Отказ
Не отвечает
ЧС
Статус меняется вручную ответственным.
8. Что хранится в карточке заявки
Обязательно:
Вся переписка (таймстемпы)
Кто отвечал
Статус
Источник
Метки (VIP / повторный клиент / риск)
Комментарии (внутренние, клиент их не видит)
9. Защита от “слива” контактов
В системе должны быть ограничения:
Фильтр отправки номеров телефона в сообщениях
Предупреждение при попытке отправить контакт
Логирование попытки
Возможность блокировки аккаунта
10. Работа с WhatsApp (важный момент)
Нужно подключать официальный WhatsApp Business API, а не обычный телефон.
Схема:
WhatsApp → API → CRM → ответственный → API → клиент
Иначе будет невозможно масштабировать и контролировать.
11. Как распределять заявки
Можно реализовать один из вариантов:
Вариант 1 — ручной
Менеджеры сами берут заявку.
Вариант 2 — авто-назначение
Если заявка с анкеты — автоматически девушке.
Если общий вход — распределение по очереди.
PM должен выбрать логику до начала разработки.
12. Контроль для собственника
В системе должно быть:
Онлайн список активных заявок
Кто отвечает
Среднее время первого ответа
Количество закрытых сделок
Конверсия в оплату
13. Что важно не делать
Нельзя использовать личные WhatsApp сотрудников
Нельзя давать сотрудникам доступ к клиентской базе
Нельзя позволять видеть чужие переписки
Нельзя хранить контакты в открытом виде
14. Минимальная архитектура
Нужно:
Telegram Bot API
WhatsApp Business API
Backend сервер
База данных (хранение заявок)
Панель управления (веб или Telegram-интерфейс)
