Архитектурная пригодность Соответствие домену задачи ★★☆☆☆☆☆☆☆☆

WordPress — движок публикации контента. Любая доменная логика (бронирование, эскроу, роли) реализуется через хуки и мета-поля поверх чужой архитектуры. Каждый новый бизнес-сценарий — отдельный плагин стороннего вендора.

 ★★★★★★★★★★ Монорепо с явными доменными модулями: Bookings, Escrow, Models, Notifications. Каждый модуль — отдельный NestJS-сервис со своей схемой БД, DTO и тестами.
Безопасность Поверхность атаки, аудит зависимостей ★☆☆☆☆☆☆☆☆☆

Каждый активный плагин выполняется с правами процесса и имеет полный доступ к БД. Нет изоляции между расширениями. Обновление одного плагина регулярно открывает уязвимость в другом.

~96% взломанных CMS-сайтов — WordPress (W3Techs, 2024). Среднемесячное число новых CVE для WP-плагинов: 40–80 штук.

 ★★★★★★★★★☆ Guards на уровне NestJS, class-validator на входе каждого эндпоинта, параметризованные запросы Drizzle. Один кодовый контур — одно ревью. Не 10/10: внешний пентест ещё не проводился.
Ролевая модель client · model · manager · admin · moderator ★★☆☆☆☆☆☆☆☆

Встроенные роли WP (subscriber / editor / administrator) не соответствуют ни одной из нужных. Добавление кастомной роли с гранулярными правами — регистрация capability, ACF-группы, хуки проверок. При обновлении ядра или плагина конфигурация ломается без предупреждения.

 ★★★★★★★★★★ JWT содержит поле role. Декоратор @Roles() на каждом эндпоинте. Отдельные зоны UI: /cabinet для клиента, /dashboard для персонала. Матрица доступа — в коде, версионируется вместе с проектом.
Схема данных и миграции Контроль изменений схемы БД ★★☆☆☆☆☆☆☆☆

Доменные данные хранятся в wp_posts + wp_postmeta — EAV без foreign keys и типизации. Изменение схемы — ручной SQL или плагин. Миграции не версионируются.

Добавить NOT NULL-колонку к таблице с 50k+ строк без даунтайма в WP невозможно — инфраструктуры для zero-downtime schema migration нет.

 ★★★★★★★★★☆ Drizzle ORM: типизированная схема в TypeScript, миграции через drizzle-kit generate/push, foreign keys, индексы. Каждое изменение схемы — PR с ревью и CI. Не 10/10: полное индекс-покрытие ещё не завершено.
Медиа и модерация Категории, видимость по роли, очередь проверки ★★☆☆☆☆☆☆☆☆

WordPress Media Library не поддерживает бизнес-категории альбомов или видимость файлов в зависимости от роли запрашивающего. Очередь модерации фото — отдельный custom post type со своим жизненным циклом, не связанным с профилем модели.

 ★★★★★★★★☆☆ Таблица media: поля albumCategory, isPublicVisible, sortOrder. API отдаёт только те файлы, которые разрешены ролью из JWT. Хранилище — MinIO S3. Не 10/10: авто-ресайз и CDN-кэширование в роадмэпе.
Бронирование и статусы Жизненный цикл под бизнес-правила ★☆☆☆☆☆☆☆☆☆

WooCommerce моделирует заказ товара, не сервисную бронь. Статусы pending → processing → completed не покрывают ни «ожидание подтверждения моделью», ни «эскроу заморожен», ни «спор открыт».

Добавление нового статуса — кастомный код внутри WooCommerce, который ломается при каждом обновлении плагина (релизы каждые 2–3 недели).

 ★★★★★★★★★☆ BookingsService: явный конечный автомат. Каждый переход — PostgreSQL-транзакция + запись в audit_events. Новый статус = строка в enum + migration. Не 10/10: экран «бронь + следующий шаг» в ЛК ещё в разработке.
Эскроу и платёжный флоу Холд · спор · выплата · возврат ★☆☆☆☆☆☆☆☆☆

WooCommerce реализует транзакцию «покупатель → продавец». Сервисный эскроу с тремя сторонами (клиент, модель, платформа), статусами удержания и аудитом — за пределами модели данных любого WP-плагина.

✕ реализовать средствами WordPress — невозможно
 ★★★★★★★★☆☆ EscrowModule: state machine, escrow_audit_events, провайдер-агностик. TON USDT — реализован. YooKassa и Stripe подключаются как отдельные адаптеры. Не 10/10: фиатный адаптер и UI разрешения споров — следующая итерация.
TON / крипто-интеграция Блокчейн-депозиты, jetton transfer, hot wallet ☆☆☆☆☆☆☆☆☆☆

PHP-экосистема не имеет библиотек для работы с TON. Фоновый polling TonAPI, парсинг JettonTransfer, идемпотентная запись депозита и подписание транзакций с Ed25519 требуют Node.js — это не слой поверх WP, это замена всего бэкенда.

✕ физически невозможно на WordPress
 ★★★★★★★★★☆ TonEscrowIndexerService: polling каждые N сек, cursor по logical_time, идемпотентность по txHash. TonHotWalletService: подписание и broadcast из admin API. Не 10/10: e2e-мониторинг индексера на mainnet ещё настраивается.
Telegram-бот Привязка аккаунта, нотификации эскроу ★☆☆☆☆☆☆☆☆☆

WP-плагины для Telegram умеют только рассылки и уведомления о заказах WooCommerce. Привязать chatId к конкретному пользователю JWT, верифицировать через /link-команду и отправлять типизированные события смены статуса эскроу — без Node.js невозможно.

✕ физически невозможно на WordPress
 ★★★★★★★★☆☆ TelegramModule (grammy, webhook): /link привязывает chatId к userId. TelegramNotifyService уведомляет обе стороны при каждом переходе эскроу. Не 10/10: inline-кнопки и шаблоны rich-сообщений — следующая итерация.
Сквозная типизация БД → API → UI — один контракт ☆☆☆☆☆☆☆☆☆☆

PHP не компилируется. TypeScript-интерфейсы на фронте — ручная копия PHP-структур. Рассинхронизация между ними проявляется в runtime на production, а не при сборке. Нет способа гарантировать контракт между слоями.

✕ сквозной type safety на WP невозможен
 ★★★★★★★★★☆ Drizzle генерирует TS-типы из схемы → NestJS DTO (class-validator) → Next.js (@escort/db). Несоответствие типов = ошибка tsc в CI, не инцидент в production. Не 10/10: часть ответов API пока типизирована как any.
Масштабирование Горизонтальный рост без архитектурных изменений ★★☆☆☆☆☆☆☆☆

PHP-процесс WordPress бутстрапится заново на каждый запрос. Горизонтальный скейл требует общего файлового хранилища (NFS или S3) и согласованного сессионного хранилища (Redis/Memcached) — всё это отдельные системы вне WP.

При кастомной бизнес-логике типичный WP падает под нагрузкой ~200 RPS без Varnish + Redis. Настройка кэш-слоя — отдельный DevOps-проект.

 ★★★★★★★★☆☆ Stateless API (JWT, без серверных сессий). Медиа на MinIO/S3. Горизонтальный скейл — replicas в docker-compose или k8s без изменения кода. Не 10/10: нагрузочное тестирование и Redis-кэш ещё не настроены.
Стабильность при обновлениях Предсказуемость релизов и зависимостей ★★☆☆☆☆☆☆☆☆

Обновление плагина или ядра WP может изменить поведение хуков, которые используют другие плагины. Цепочка из 10–15 зависимостей — это 10–15 независимых векторов регрессии при каждом обновлении.

Проекты на WP с кастомной логикой тратят 40–60% dev-бюджета на поддержку совместимости, а не на развитие продукта (Smashing Magazine, 2023).

 ★★★★★★★★★☆ Монорепо: все зависимости под контролем команды. Изменение проходит через PR, CI и тесты. Нет сторонних вендоров с независимым циклом релизов. Не 10/10: e2e-покрытие ~30% критических путей, растёт итерационно.
Совокупная стоимость владения Скрытые расходы vs стоимость разработки ★★★☆☆☆☆☆☆☆

Хостинг и базовые плагины дёшевы. Но первый нестандартный сценарий (эскроу, кастомные роли, крипта) запускает нарастающий технический долг: кастомные плагины, поддержка совместимости, регулярные аварийные фиксы.

Типичная стоимость аварийного восстановления после обновления у платёжного WP-проекта: от $2 000. Ежегодные лицензии плагинов: $500–2 000.

 ★★★★★★★★☆☆ Нет лицензионных платежей. Каждый час разработки вкладывается в бизнес-логику, а не в поддержку совместимости. Кодовая база — актив компании. Не 10/10: выше порог входа для нового разработчика по сравнению с WP.
Резюме: WordPress оптимален для контентных сайтов и стандартных интернет-магазинов. Для платформы с сервисным эскроу, криптовалютными депозитами, 5-уровневой ролевой моделью и Telegram-ботом — это не тот класс инструмента. Ряд критических функций невозможно реализовать средствами WP без фактической замены всего бэкенда на отдельный Node.js-сервис, что полностью нивелирует любую экономию на старте.

Невозможно реализовать на WordPress

TON / крипто-эскроу

Нативный парсинг JettonTransfer с TonAPI, атомарная запись депозита в PostgreSQL-транзакции, cursor по logical_time — в WordPress нет PHP-библиотек для TON и runtime для long-lived background services.

JWT с доменными ролями

WordPress использует cookie-сессии и встроенные capabilities. Выдать JWT с полем role: 'model' | 'client' | 'manager' и верифицировать его через декоратор @Roles() — архитектурно невозможно.

Telegram-бот в монорепо

Привязка Telegram-аккаунта к JWT-пользователю, webhook-режим, типизированные уведомления при переходе эскроу — WordPress не имеет NestJS-провайдеров и фоновых сервисов.

Stateful эскроу (state machine)

Автомат draft → pending_payment → escrow_funded → confirmed → completed / disputed с атомарными переходами и аудит-таблицей — WooCommerce предоставляет «корзину», не сервисный эскроу.

Сквозной type safety БД → UI

Drizzle генерирует TypeScript-типы из схемы — NestJS и Next.js используют их напрямую. В WordPress/PHP этот сквозной type-safety физически недостижим: ошибка типа = runtime-ошибка в production.

Индексер блокчейна

Фоновый NestJS-сервис, каждые N секунд опрашивающий TonAPI и идемпотентно записывающий депозиты по txHash — WordPress не поддерживает long-running Node.js-процессы.

Кастомный медиа-пайплайн

MinIO S3 с категориями альбомов (portfolio / vip / elite), видимостью файлов по роли из JWT и CDN-прокси — WordPress Media Library не поддерживает доменную логику видимости без полного переписывания.

Hot Wallet для TON-выплат

Подписание TON-транзакций с mnemonic прямо из admin API-эндпоинта — PHP не имеет зрелой экосистемы для TON SDK и Ed25519-криптографии уровня production.
