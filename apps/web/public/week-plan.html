Аудит + недельный план

План работ на неделю

Сгенерировано по состоянию репозитория escort-platform (monorepo: apps/api, apps/web, packages/db). Оценка Phase 1 CMS ≈ 65% (см. COMPREHENSIVE_AUDIT_AND_PLAN.md, CLAUDE.md).

Краткий аудит структуры

  • Backend: NestJS (apps/api), Swagger /api/docs, модули (auth, models, moderation, reviews, media и др.), JWT, интеграция с PostgreSQL через Drizzle.
  • База: packages/db — схемы Drizzle, миграции; Docker Compose dev: Postgres, Redis, MinIO, Mailhog.
  • Frontend: Next.js App Router (apps/web), публичные страницы (главная, каталог, профиль модели, контакты, помощь, логин), дашборд с темой wp-admin, BFF-вызовы к API.
  • Уже сделано недавно: отзывы с модерацией, очередь модерации, subscriptionTier в JWT, публичная шапка (fixed, стеклянные капсулы), унификация золотых кнопок под стиль «Вход» (.btn-primary + globals.css).
  • Пробелы (из роадмэпа): видимость фото поштучно, альбомы, fade-slider, water shader для гостей, booking UI, оплаты — в основном не завершены.
  • Вне основного контура: каталоги enso-develop, NON_PROJECT, prototypes — не смешивать с прод-приложением без явного решения.

Ниже — календарь на 7 дней с детализацией по задачам. Время внутри дня ориентировочное; порядок можно менять по зависимостям.

День 1 — Стабилизация и дизайн-система кнопок

  1. Утро (2–3 ч): Прогон локального стека: docker compose -f docker-compose.dev.yml up -d, API :3000, Web :3001; проверка логов предупреждений MinIO/Postgres.
  2. Утро: Smoke-тест публичных маршрутов: /, /models, /models/[slug], /contacts, /help, /login — формы, шапка, мобильное меню.
  3. День: Визуальный QA кнопок: все btn-primary = золотая стеклянная капсула как «Вход»; btn-secondary = капсула со стеклянной обводкой (уже в globals.css).
  4. День: Регрессия доступности: focus-visible на ссылках/кнопках, контраст подписей на герое.
  5. Вечер (1–2 ч): Зафиксировать заметки в issue/доке: известные баги, скриншоты расхождений с DESIGN.md.

День 2 — Видимость изображений (API + контракт)

  1. Утро: Спецификация поля(ей) в БД: флаг или enum «скрыто/видимо» на уровне медиа записи модели; согласовать с существующей таблицей медиа в Drizzle.
  2. Утро: Миграция + обновление entity/DTO в NestJS; права: модель/менеджер могут менять, публичная выдача фильтрует скрытые.
  3. День: Endpoints: PATCH/PUT для обновления видимости; GET каталога и профиля — только видимые для анонима; staff — полный набор.
  4. День: Юнит-тесты или e2e на критический путь (создать медиа → скрыть → публичный GET не отдаёт).
  5. Вечер: Обновить Swagger и краткий фрагмент в platform-blueprint.html при необходимости.

День 3 — Видимость изображений (Web)

  1. Утро: Типы/клиент API на фронте; хуки загрузки профиля и списка медиа с новым полем.
  2. День: UI в dashboard/.../edit: переключатель/иконка глаза на каждом фото, оптимистичное или подтверждаемое сохранение.
  3. День: Публичная карточка и страница модели: гарантированно не рендерить скрытые URL; проверка кэша Next/Image.
  4. Вечер: Ручной тест под ролями client/model/manager; исправление edge cases (главное фото скрыто).

День 4 — Альбомы / категории фото

  1. Утро: Модель данных: альбом (id, modelId, title, sort) + связь media→album (nullable для обратной совместимости).
  2. Утро: Миграции Drizzle; сервисы NestJS: CRUD альбомов, перенос фото между альбомами.
  3. День: API: список альбомов с медиа, лимиты количества (согласовать с продуктом).
  4. День: Дашборд: простой UI списка альбомов + drag-sort опционально (или кнопки вверх/вниз).
  5. Вечер: Публичный профиль: секции или табы по альбомам (минимальная версия — один ряд карточек на альбом).

День 5 — Fade slider и полировка профиля

  1. Утро: Вынести/создать компонент fade-слайдера (доступность, prefers-reduced-motion, автоплей опционально).
  2. День: Интеграция в публичный профиль модели (галерея/хедер по решению UX); fallback на текущий вид при одном фото.
  3. День: Регрессия Lighthouse на странице профиля (CLS, LCP); lazy-loading.
  4. Вечер: Мелкие UX: подписи, жесты на мобильных, состояние загрузки.

День 6 — Water shader / герой и booking UI (черновик)

  1. Утро: Аудит WaterSurface: режим photoOnly vs WebGL; критерии когда включать шейдер (не ломать incognito).
  2. День: Либо тонкая настройка существующего шейдера, либо отложить с флагом feature — зафиксировать в коде env/настройке.
  3. День: Начать guest-facing booking: макет шага 1 (выбор даты/длительности) на отдельном маршруте или модалке; заглушка submit.
  4. Вечер: Согласование с API бронирований (что уже есть в NestJS) — список endpoints и пробелы.

День 7 — Интеграция, тесты, релиз-ноты

  1. Утро: Просмотр открытых TODO в дашборде (золотые кнопки там пока отдельные классы — решить: постепенная унификация vs оставить wp-admin стиль).
  2. День: Прогон npm run build в apps/web и apps/api; исправление ошибок TypeScript/ESLint критичных.
  3. День: Минимальный e2e или чеклист QA по сценарию «гость → каталог → профиль → контакт».
  4. Вечер: Обновить блок «Роадмэп» в platform-blueprint.html статусами; короткий CHANGELOG или запись в COMPREHENSIVE_AUDIT_AND_PLAN.md.
  5. Вечер: Подготовка к деплою: env, миграции на стенде, бэкап БД.
Исходный роадмэп этапов 0–9: platform-blueprint.html → вкладка «Роадмэп». Файл плана: apps/web/public/week-plan.html.
