Раздел

Оплата — черновой поток (пример с Telegram)

Сценарий для обсуждения: гость на сайте выбирает модель и жмёт «Связаться». Куда его ведут — зависит от продукта: часто это deep link в Telegram (t.me/YourBot?start=…), где в параметре start закодированы model_id и одноразовый nonce (или короткий код с сервера). Бот принимает /start, дергает API платформы (webhook или HTTPS): создаётся или подтверждается намерение брони, при необходимости фиксируется оплата у провайдера. Платформа выдаёт гостю короткоживущий токен (JWT или opaque id) и/или ссылку «вернуться на сайт с ?token=» — так веб и бот сходятся на одной сущности брони без раскрытия пароля от аккаунта. Ниже — упрощённая DFD; детали алгоритма (эскроу, статусы, PCI) вынесены на вкладку «Эскроу» и в бэклог.

0 Продукт и фундамент репозитория

Выбор стека, каркас монорепо, локальная инфраструктура разработчика.

  • Стек: Next.js 15 (App Router), NestJS, TypeScript, Drizzle ORM, PostgreSQL.
  • Монорепо: Turborepo / npm workspaces, apps/api, apps/web, packages/db.
  • Docker dev: Postgres, Redis, MinIO, Mailhog (docker-compose.dev.yml).
  • Документация старта: README, CLAUDE.md, health/Swagger.
  • CI/CD production: пайплайн под prod, секреты вне .env — не доведено.

1 Ядро данных и API (доменная модель)

Сущности агентства: кто с кем взаимодействует на бэкенде.

  • Auth: JWT (access/refresh), bcrypt, роли admin / manager / model / client.
  • Пользователи и профили: users, привязка к модели/клиенту.
  • Модели: анкеты, публикация, связь с медиа.
  • Клиенты, бронирования: state machine в API.
  • Эскроу: сущности транзакций в БД/API (без живого провайдера).
  • Отзывы: создание staff, модерация, публичная выдача одобренных.
  • Чёрный список, сессии, аудит (база): таблицы и эндпоинты по мере модуля.
  • Политика доступа: subscriptionTier у пользователя и в JWT для правил отзывов/доступа.
  • Модерация единой очередью: /models/moderation/* (профили, медиа, отзывы).

2 CMS: внутренняя админка контента

Дашборд для менеджеров и моделей: анкеты, медиа, настройки.

  • Вход и оболочка: login, admin-login, AuthProvider, дашборд-лейаут.
  • Модели CMS: список, создание, редактирование, просмотр, страница фото.
  • Загрузка медиа: presigned, базовый upload UI.
  • Видимость фото: show/hide на публичной карточке, per-file — не реализовано.
  • Альбомы / категории (Portfolio / VIP / Elite), сортировка drag-and-drop, массовые действия.
  • Валидация форм и понятные ошибки — частично.
  • Страница модерации в дашборде (очередь из API).
  • Тема wp-admin дашборда, типографика настроек — в работе/улучшено.

3 Дизайн-система и публичная витрина

Единый визуальный язык и «витрина» для гостя: каталог и карточка модели.

  • DESIGN.md: шрифты (Unbounded + Inter), палитра, сетка, motion.
  • Публичный каталог /models и карточка /models/[slug].
  • Fade slider фона на карточке (кроссфейд, интервал, lazy-load).
  • Water shader оверлей (Three.js), fallback на мобилках.
  • Галерея lightbox и полировка мобильной вёрстки публичных страниц.
  • OG / соц. превью и шаринг.

4 Сценарий гостя: контакт, заявка, бронь в UI

От просмотра анкеты до согласованной брони в интерфейсе (не только API).

  • Кнопка «Связаться» / контакт: UX и интеграция с ботом или формой — в blueprint (DFD), в продукте не замкнуто.
  • Форма запроса брони с публичной страницы (дата, время, услуга).
  • Кабинет гостя (опционально): статус заявки, история.
  • Кабинет менеджера: список броней и смена статусов — смотреть покрытие UI в дашборде.

5 Платежи, эскроу и финансовый контур

Реальные деньги: провайдер, вебхуки, холд и разблокировка.

  • Черновой DFD оплаты (Telegram / токен брони) — на вкладке «Финансы» blueprint.
  • Интеграция провайдера (YooKassa, Cryptomus или выбранный).
  • Эскроу end-to-end: холд, подтверждение встречи, выплата, отмены, споры.
  • PCI / compliance и аудит платёжных событий.

6 Каналы связи и CRM

Операционка агентства: переписки и напоминания.

  • Telegram-бот и deep link из сайта (см. DFD «Оплата»).
  • WhatsApp / единый inbox для менеджеров.
  • Email: Mailhog в dev; транзакционные письма в prod — не подключены.
  • Шаблоны уведомлений (бронь, оплата, модерация).

7 Безопасность, доверие, соответствие

Защита данных, учёток и контента.

  • JWT / guards: в dev часть защиты ослаблена — закрыть перед prod.
  • Rate limiting на auth и публичные формы.
  • Верификация email, сброс пароля, 2FA для админов.
  • Инвалидация сессий при смене пароля; политика refresh.
  • Санитизация пользовательского контента (XSS), CSP.
  • Расширенный audit log чувствительных действий.
  • Юридические страницы и (при необходимости) экспорт данных пользователя.

8 Качество, тесты, наблюдаемость

Уверенность в релизах и диагностика в бою.

  • Ручной регресс: вкладка «Тестирование» в blueprint + чеклист в аудите.
  • Unit / integration тесты API и критичных сервисов.
  • E2E (Playwright и т.п.) в CI.
  • Error boundaries, loading states, retry по всему веб-приложению.
  • Логирование, метрики, алерты для production.
  • Бэкапы БД и runbooks восстановления.

9 Рост: аналитика, монетизация, расширенная админка

После стабильного ядра — масштабирование продукта и операций.

  • Дашборд аналитики (воронка, конверсии, выручка).
  • VIP / Elite gating контента по подписке или роли.
  • Внешний IdP (Clerk и др.) — опционально вместо самописного JWT.
  • Расширенная админка: отчёты, массовые операции, feature flags.

DFD

Выбранный поток

Справка · DFD

Формы блоков (Гейн–Сарсон)

  • Процесс — круг (на поле: номер D + имя). Преобразование данных.
  • Внешняя сущность — прямоугольник. Источник или приёмник данных вне контура диаграммы (гость, платёжная система, другой сервис).
  • Хранилище данных — открытый прямоугольник с «боковыми» линиями. Пассивное хранение (БД, файлы, кэш).

Карточки в палитре можно перетащить на сетку или добавить процесс кнопкой «+ Процесс на поле».

Цвета линий

Новый поток наследует цвет от узла-источника (его «провод»). У новых узлов цвет по очереди циклический:

Бирюзовый — внешний / клиентский контур (часто гость, публичный фронт).
Оранжевый — действия, загрузки, исходящие операции.
Зелёный — положительное ветвление, успех, «да».
Синий — ядро, правила, шлюзы, решения платформы.
Розовый — внутренние потоки staff, модерация, очереди.
Серый — нейтральный поток по умолчанию.

В шаблоне «ветвление модерации» отдельные рёбра могут быть заданы вручную: чёрный пунктир до модерации, зелёное «да», красное «нет»; стрелки на концах совпадают с цветом линии.

Потоки данных

  • Связь тянется от ● справа (выход процесса) к ● слева (вход). Стрелка — направление данных.
  • Клик по линии — выбор потока, подпись в панели слева.
  • Двойной клик по линии — удалить связь.
  • Esc — снять выбор потока (если открыта эта справка — сначала закроется она).

Холст

  • Колёсико мыши — масштаб к курсору.
  • ПКМ + перетаскивание — панорама.
  • ЛКМ по узлу — перетаскивание блока.

Подсветка рёбер

  • Наведение на линию — бегущий пунктир по направлению к целевому блоку.
  • Наведение на блок — то же для всех исходящих из него рёбер.

Шаблон «отзывы · доступ · модерация»

Готовая DFD: гость без просмотра; модель — только отзывы о себе; менеджер — только по своим моделям; подписчики с уровнями; отзыв только после состоявшейся встречи; общая модерация.

Шаблон «админ / клиент · ветвление модерации»

«Админ» → «Оставить отзыв» напрямую; «Клиент» → «Модерация»: до узла модерации ребро чёрное пунктирное, после решения — да зелёное, нет красное.
