Эскроу и безопасная сделка

Техническое руководство для разработчиков монорепо escort-platform (apps/api, apps/web, packages/db). Юридические и тарифные условия — только из договора и актуальной документации выбранного банка. Примеры URL и тел запросов ниже — иллюстративные; перед кодированием сверяйтесь с официальным OpenAPI / PDF от Т-Банка, ЮKassa или Сбербанка.

Что предлагает Т-Банк для эскроу

Т-Банк (ранее Tinkoff) в контурe Tinkoff Business API предлагает продукт уровня «Безопасная сделка» (терминология и доступность методов — по договору и документации на момент подключения). Типичные возможности продукта:

  • создание сделки: сумма, валюта, идентификация сторон, условия исполнения;
  • хранение средств на выделенном эскроу-контуре до наступления условий;
  • подтверждение исполнения (в т.ч. двустороннее, по сценарию банка);
  • завершение сделки — автоматически по правилам или вручную (роли из договора);
  • возврат при отмене / истечении срока / спорном сценарии (по регламенту банка).

Интеграция идёт через REST (и вебхуки). Точные пути, заголовки авторизации и формат подписи вебхука — только из актуальной спецификации банка, не из этого черновика.

Контекст: четыре комиссии

В договоре обсуждаются до трёх комиссионных потоков в стандартном продукте; бизнесу нужны четыре (платформа, банк, дев, менеджер). Банк может зафиксировать четвёртую линию в договоре, а реализацию четвёртого распределения в ПО оставить на платформе — это должно отразиться в фазах 3–4 (маппинг сумм, отчётность, внутренний сплит при варианте B из бизнес-дока).

Связь с текущим кодом репозитория

Уже есть таблица escrow_transactions (packages/db/src/schema/escrow.ts): связь 1:1 с bookings, поля paymentProvider (yookassa | cryptomus | manual), paymentProviderRef, статусы pending_funding → funded → … Сервис EscrowService (apps/api/src/escrow/) создаёт запись и меняет статусы. Задача интеграции Т-Банка — не плодить вторую модель «сделки» без нужды, а расширить провайдер и внешние идентификаторы.

Сценарий бронирования простыми словами

Ниже — как обычно договариваются с банком и что кладём в код. Формулировки «покупатель / продавец» — это роли из договора с банком, их нужно один раз сопоставить с нашими сущностями (клиент, модель, платформа).

Что значит funded (зачислено / залочено)

pending · ожидает в БД: pending_funding
>>>
стало
>>>
funded зачислено, скажем так — на эскроу, деньги залочены

Не путать с «встреча прошла». Статус funded в escrow_transactions значит только одно:

  • клиент уже оплатил (или деньги захолдированы — зарезервированы) по этой брони;
  • банк подтвердил, что сумма лежит на эскроу (на «безопасной сделке»), а не у клиента на карте как свободные деньги;
  • деньги ещё никому не «выплачены» — ни модели, ни платформе: они заблокированы до правил сделки.

Коротко: funded = деньги в сейфе по этой брони, услуга ещё не считается оказанной. Дальше по сценарию кто-то подтверждает факт встречи (или срабатывает таймаут / спор) — и только потом идёт разблокировка (released) или возврат (refunded).

У брони в БД есть свой статус, например escrow_funded — это про жизнь брони; у эскроу — funded — про деньги. Их переводят согласованно (см. шаги ниже).

Кто «покупатель», кто «продавец» в терминах API банка

Покупатель (buyer, плательщик) — почти всегда это клиент, который платит за бронь: в нашей базе это пользователь bookings.clientId. Именно он вводит карту и списывает/блокирует сумму.

Продавец (seller) — тот, с кем банк формально заключает «продажу» в рамках безопасной сделки. Здесь два рабочих варианта (выбирается с юристом и банком, не программистом в одиночку):

  • Вариант 1 (часто у агентских платформ): продавец = платформа (юрлицо оператора). Модель потом получает свою долю уже после разблокировки эскроу — по внутренним договорам и выплатам.
  • Вариант 2: продавец = модель или её ИП/самозанятый (если банк и договор это допускают и есть счёт получателя).

Для банка важно, кто указан в полях сделки; для нас в коде важно, чтобы bookingId и суммы совпадали с тем, что ушло в API.

Примерная последовательность (от брони до выплаты) — 11 шагов в одной схеме

Та же логика, что у цепочки pending >>> стало >>> funded, только от начала до конца успешного сценария. Между блоками — >>> («далее произошло»). Ветка «спор / возврат» — отдельно, не в этой линейке.

1Броньклиент создал · bookings
>>>
2Эскроу ждётpending_funding
>>>
3Окно оплатывиджет / редирект банка
>>>
4Клиент платиткарта, СБП…
>>>
5Банк принялхолд / эскроу-счёт
>>>
6Вебхук / ответподтверждение в API
>>>
7Эскроу fundedденьги залочены
>>>
8Броньescrow_funded
>>>
9Встречапо startTime
>>>
10Услуга ОКменеджер / 2 стороны / таймаут
>>>
11Конец сделкиreleased · completed · сплит

Шаги 7–8 часто делают в одном обработчике вебхука: сначала фиксируем funded у эскроу, сразу же обновляем бронь на escrow_funded. Шаг 11 — вызов API банка на завершение / выплату и запись released + completed, затем учёт комиссий (мы / банк / дев / менеджер).

Если встреча сорвалась или спор (не линейное продолжение): не выполняем шаги 10–11 по выплате; инициируем возврат по банку → refunded / partially_refunded, бронь cancelled / refunded / disputed — как в продукте.

Важно для разработки: перевод в funded делать только по событию банка «деньги на эскроу», а не по нажатию «забронировать» у клиента. Подтверждение услуги и released — отдельная ветка логики.

Пошаговый план внедрения (для программиста)

Фаза 0 — Входные данные от банка и продукта

  1. Получить от менеджера Т-Банка: ссылку на OpenAPI / Postman, описание статусов сделки, список вебхуков, требования к IP whitelist, формат подписи (например заголовок вроде X-Tinkoff-Signature — имя уточнять).
  2. Зафиксировать sandbox base URL и production base URL (в доке они различаются).
  3. Согласовать с бизнесом сценарий из блока «Сценарий бронирования простыми словами» выше: роли buyer/seller для банка, момент перехода в funded (только по событию банка), способ подтверждения услуги и правила спора/возврата.
  4. Выбрать стратегию комиссий (4 потока в API банка vs 3 + внутренний сплит менеджера) — от этого зависит тело запроса создания сделки.

Фаза 1 — Конфигурация и секреты

  1. Добавить в .env.example (и прод-секреты): TBANK_API_URL, TBANK_CLIENT_ID, TBANK_CLIENT_SECRET или иной способ авторизации по доке; TBANK_WEBHOOK_SECRET / ключ проверки подписи; TBANK_SANDBOX=true|false.
  2. В apps/api/src/config/validation.schema.ts (или аналог) добавить опциональные поля и валидацию, чтобы прод не стартовал с полупустым платёжным конфигом, если модуль включён.
  3. Нигде не коммитить секреты; для CI — переменные окружения хостинга.

Фаза 2 — Миграция БД (Drizzle)

  1. Расширить union paymentProvider значением 'tbank' (миграция + обновление TypeScript-типов в схеме).
  2. В paymentProviderRef хранить основной внешний ID сделки в Т-Банке (как возвращает API).
  3. Рекомендуется добавить поля (отдельная миграция): например tbank_deal_id (если ref недостаточно), last_webhook_at, last_known_external_status — чтобы расследовать расхождения без парсинга истории.
  4. Опционально таблица escrow_webhook_events: id, escrowTransactionId, eventId (уникальный от банка), payload (jsonb), receivedAt — для идемпотентности и аудита.
  5. Согласовать маппинг статусов банка → поле escrow_transactions.status и дополнение stateHistory (уже есть jsonb) при каждом переходе.

Фаза 3 — Слой API (NestJS)

  1. Создать модуль, например apps/api/src/payments/tbank/tbank.module.ts, или расширить EscrowModule, если хотите меньше сущностей — на усмотрение команды.
  2. HTTP-клиент: в проекте сейчас нет @nestjs/axios; используйте нативный fetch (Node 22) или добавьте axios только при необходимости. Обёртка с таймаутом, ретраями только для идемпотентных GET и осторожно.
  3. Сервис TBankEscrowService (имя условное):
    • getAccessToken() — кэш в Redis с TTL чуть меньше срока жизни токена (ключ вида tbank:oauth:access).
    • createDealForBooking(bookingId) — загрузить booking, сумму, валюту; вызвать API банка; записать escrow_transactions с paymentProvider: 'tbank', paymentProviderRef; статус pending_funding.
    • handleWebhook(rawBody, headers) — проверка подписи по алгоритму банка; идемпотентность по event_id (Redis SET NX или уникальный индекс в escrow_webhook_events); обновление строки эскроу и stateHistory.
  4. Контроллер публичного вебхука: маршрут вида POST /webhooks/tbank (глобальный префикс уточнить в main.ts), сырое тело для проверки подписи (raw body middleware в Nest).
  5. Внутренние эндпоинты (JWT + роли): создание эскроу по брони, ручной триггер «синхронизировать статус с банком», админский возврат — по политике продукта.

Пример структуры сервиса (псевдокод, URL вымышленные)

// apps/api/src/payments/tbank/tbank-escrow.service.ts
@Injectable()
export class TBankEscrowService {
  // baseUrl из конфига: sandbox vs prod
  async createEscrowDeal(input: { bookingId: string; amount: string; currency: string }) {
    const token = await this.getCachedAccessToken();
    const res = await fetch(`${this.baseUrl}/v1/secure-deals`, {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${token}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        amount: input.amount,
        currency: input.currency,
        // поля сторон и callback_url — строго по доке банка
        callback_url: `${this.publicApiBase}/webhooks/tbank`,
      }),
    });
    const data = await res.json();
    // insert escrow_transactions: paymentProvider 'tbank', paymentProviderRef = data.deal_id
    return data;
  }

  async handleWebhook(payload: unknown, headers: Record<string, string>) {
    this.assertValidSignature(payload, headers);
    const eventId = extractEventId(payload);
    if (await this.wasWebhookProcessed(eventId)) return;
    await this.applyStatusToEscrow(payload);
    await this.markWebhookProcessed(eventId);
  }
}

Фаза 4 — Маппинг статусов и таймауты

  1. Таблица соответствия: каждый статус Т-Банка → один из pending_funding | funded | hold_period | released | refunded | disputed_hold | partially_refunded. Неоднозначные промежуточные статусы банка хранить в last_known_external_status.
  2. @nestjs/schedule или внешний cron: задачи «истёк срок холда», «автоосвобождение», «напоминание о подтверждении» — с вызовом API банка или только внутренней сменой состояния, в зависимости от регламента.
  3. Все переходы, инициированные cron, писать в stateHistory.

Фаза 5 — Web (Next.js 15)

  1. Маршрут гостя/клиента, например apps/web/app/bookings/[id]/pay/page.tsx или /deals/[id]: серверный fetch к API по cookie/JWT; отображение EscrowStatusCard.
  2. Если банк отдаёт ссылку на оплату / виджет — встроить по инструкции банка; иначе редирект на hosted checkout.
  3. После оплаты пользователь возвращается на return_url — страница должна запросить актуальный статус у API (не доверять только query-string).
  4. Для менеджера в дашборде — кнопки «подтвердить оказание услуги» / «открыть спор», вызывающие защищённые API Nest.
  5. Уведомления в реальном времени (WebSocket/SSE) — опционально, можно начать с polling статуса раз в N секунд на странице сделки.
// apps/web/app/bookings/[id]/pay/page.tsx — идея
export default async function BookingPayPage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const deal = await serverFetchEscrowByBooking(id);
  return (
    <div className="max-w-2xl mx-auto p-6">

Фаза 6 — Инфраструктура и прод

  1. Публичный HTTPS endpoint для вебхуков (Nginx → Nest :3000); проверить, что тело запроса не модифицируется (gzip/buffer).
  2. Redis уже в compose — использовать для OAuth-кэша и идемпотентности вебхуков.
  3. Логи: не писать PAN/CVV; маскировать персональные данные.
  4. Алерты: рост 4xx/5xx на вебхуке, расхождение суммы в webhook и в БД, зависшие pending_funding старше N часов.

До трёх комиссионных потоков в стандартном продукте; нужно четыре (платформа, банк, дев, менеджер). Банк может прописать четвёртую линию в договоре, реализацию в ПО делает платформа. ЮKassa — кратко о продукте

ЮKassa (Яндекс/YooMoney) — популярный эквайринг в РФ. Для сценариев, близких к эскроу, обычно используют:

  • Двухстадийная оплата (холд) — списание после подтверждения capture;
  • Сплитование — распределение между получателями (при подключённом мультисплите и договоре).

В схеме БД репозитория поле paymentProvider: 'yookassa' уже предусмотрено — интеграция может быть первым шагом до Т-Банка или резервным каналом.

Сравнение с Т-Банком (для выбора)

КритерийТ-Банк «Безопасная сделка»ЮKassa
МодельНативная «сделка» под условное удержаниеПлатежи + холд + опционально сплит
ОнбордингКорпоративный контур T-Business, отдельные согласованияОбычно быстрее для ИП/ООО на классический эквайринг
МультиполучателиПод договор (в т.ч. обсуждение 4-й комиссии)Сплит по правилам ЮKassa и договору
Единый стек интеграцииНовый адаптер tbankТот же Nest + вебхуки + Redis; другой формат API

Когда выбирать ЮKassa: нужен быстрый холд + capture без тяжёлого корпоративного контура; приемлема модель «не отдельный эскроу-счёт, а платёж с отложенным списанием». Когда Т-Банк: требуется именно продуктовая «безопасная сделка» и договорённости под многостороннее распределение.

Технические шаги (тот же каркас, что у Т-Банка)

  1. Ключи YOOKASSA_SHOP_ID, YOOKASSA_SECRET_KEY в .env.
  2. Создание платежа с capture: false для холда; confirmation.return_url на Next.
  3. Вебхуки ЮKassa — проверка IP и Basic auth / секрет; идемпотентность по event.id.
  4. Маппинг на escrow_transactions и переходы статусов; capture при подтверждении услуги, refund при отмене.

Сбербанк Эквайринг / API

У Сбербанка несколько продуктов (интернет-эквайринг, Сбербанк API, линки на оплату). Классический путь «как холд» — двухстадийная оплата: авторизация (холд) → подтверждение (списание) или отмена.

Отдельного публичного аналога «безопасной сделки с несколькими участниками» в одном API может не быть; мультираспределение часто решается бухгалтерски и договорами между юрлицами поверх эквайринга.

Когда Сбербанк: уже есть действующий эквайринг Сбера, единый банк-контур для компании, готовность к более тяжёлому онбордингу API. Архитектура Nest + Drizzle + Redis остаётся той же: адаптер провайдера, вебхуки, маппинг статусов.

Роль Telegram

Ниже — две разные архитектуры. В первой деньги по-прежнему у банка/эквайринга, бот — интерфейс и уведомления. Во второй банк не участвует в расчётах: используется крипто-эскроу (ончейн или кастодийный кошелёк); это другой правовой и операционный контур — читайте предупреждения.

Формулировка «бот с деньгами на эскроу» в буквальном виде неверна: у Bot API нет счёта для хранения рублей. Деньги либо у банка-провайдера, либо в блокчейне на адресе/в контракте; бот только показывает адрес, кнопки и статусы.

Схема 1 — Telegram + банк (или ЮKassa): от брони до выплаты

Эскроу в юридическом смысле обеспечивает платёжный провайдер. Ваш Nest принимает вебхуки и ведёт escrow_transactions. Telegram — UX.

Схема 2 — без банка: «мы», клиент и Telegram (крипто-эскроу)

Не юридическая консультация. Приём и передача фиата (рублей) без лицензии может квалифицироваться как платёжная деятельность (ЦБ РФ и др.). Работа с криптоактивами в РФ регулируется отдельно (закон о ЦФА/ограничения, налоги, 115-ФЗ, санкции). Любая модель «мы — гарант» — репутационные, уголовные и гражданские риски при ошибке или мошенничестве. Перед запуском — юрист и compliance; для массовых расчётов в РФ обычно безопаснее банк/ЮKassa (схема 1).

Идея: стороны не доверяют друг другу напрямую, но согласны доверить разблокировку суммы правилам смарт-контракта или мультисиг-кошелька, где одна доля подписей у платформы (арбитр). Бот не хранит USDT — токены на адресе в сети; бот шлёт QR, кнопки «я перевёл», «подписать выплату».

Мультисиг 2-of-3 (логика, не готовый API)

Три стороны владеют долей подписи: например клиент, модель/продавец, платформа. Чтобы потратить средства с общего адреса, нужны две подписи из трёх. Типичные сценарии:

  • Сделка ОК: подписали клиент + продавец → выплата продавцу (платформа не подписывает).
  • Спор: платформа + «победившая» сторона по правилам продукта → выплата туда, куда решил арбитраж.

На практике «генерировать три ключевые пары на сервере и шифровать две клиентским» из примеров в сети — опасный антипаттерн: сервер касался приватных ключей клиента. Лучше: смарт-контракт с явными ролями, или кастодий с аппаратным модулем (HSM/KMS) только для ключа платформы, а клиент подписывает в Mini App через свой кошелёк (WalletConnect и т.д.) — отдельный тяжёлый фронт.

Пополнение эскроу (бот)

  1. API создаёт сделку и фиксирует ожидаемую сумму и уникальный адрес (или один пуловый адрес + memo/payment_id — тогда риск перепутать платежи; обычно лучше уникальный депозит на сделку).
  2. Бот отправляет текст адреса и QR (как в примерах с replyWithPhoto в Telegraf/Nest).
  3. Кнопка «Я отправил» запускает опрос индексатора (TronGrid, свой воркер по блокам): совпадение суммы, подтверждений сети, правильный контракт USDT.
  4. При успехе — тот же переход, что у банка: pending_funding → funded в вашей таблице (можно завести payment_provider: 'tron_usdt' или отдельную таблицу crypto_escrow_deposits, чтобы не смешивать с банковским эскроу без миграционной стратегии).

Освобождение средств

  1. Пользователь в TG нажимает «Подтвердить выплату» → бэкенд проверяет роль и что сделка в статусе funded.
  2. Собирается частично подписанная транзакция (в зависимости от стека) или вызывается метод смарт-контракта release().
  3. Когда набор подписей достигнут (2-of-3 или правило контракта), воркер broadcast tx в сеть.
  4. Сохранить txid, дождаться финализации, обновить статус released / completed, уведомить ботом обе стороны.
  5. Идемпотентность: как у вебхуков — один и тот же «релиз» не выполнять дважды (Redis lock по dealId).

Споры и арбитраж

  • Отдельный статус в БД (disputed), заморозка кнопок «release» для обычных пользователей.
  • Уведомление в админ-канал или второй бот с telegram_user_id дежурных.
  • Решение админа → серверная подпись платформы + вторая подпись по сценарию (или только контракт arbitrate(recipient) с ролью owner — по дизайну).

Данные в БД (Drizzle) — отдельно от банковского эскроу

Не обязательно ломать текущую escrow_transactions. Варианты: (а) расширить провайдером crypto_tron и полями escrow_address, funding_tx_hash, release_tx_hash; (б) новая таблица crypto_escrow_deals + crypto_escrow_signatures / события ончейн. Обязательны: chain_id / сеть, token_contract, лимиты подтверждений, привязка к booking_id.

Ключи и секреты

  • Ключ арбитра / сервиса — только в Vault, KMS, HSM, аудит доступа; не в .env на проде для сырого hex.
  • Отдельные воркеры с минимальными правами на подпись.
  • Клиентские приватные ключи не хранить на своём диске в открытом виде; если нужна подпись клиента — кошелёк пользователя или одноразовые схемы с явным согласием.

Фиатный мост (рубли)

Если клиент вводит рубли на расчётный счёт ИП/ООО, а вы конвертируете в USDT — это уже обмен + возможная платёжная деятельность. Делать только через лицензированных партнёров и по договору с юристом. Прямой «прими рубль на карту и зачисли в эскроу» без лицензии — высокий риск.

Три рабочих модели

МодельЧто делает TelegramГде настоящие деньги
A. Бот = уведомления + ссылки Пишет «бронь создана», «оплатите», «эскроу funded», шлёт кнопку-ссылку на https://ваш-сайт/…/pay Веб-оплата на сайте через банк/ЮKassa — как без Telegram

Для РФ чаще всего начинают с модели A или C: меньше ограничений по провайдерам, чем у встроенных Telegram Payments. Модель B имеет смысл, если у вас есть подходящий provider token и одобренный сценарий у Telegram/провайдера.

Реализация схемы 1: бот и вебхук (пошагово)

Ниже — инженерный чеклист для Telegram как канала к уже подключённому банку/ЮKassa.

1. Регистрация и секреты

  1. В Telegram открыть @BotFather → /newbot → получить HTTP API token.
  2. Сохранить в .env: TELEGRAM_BOT_TOKEN, не коммитить.
  3. Опционально задать описание, команды (/setcommands) — для UX.

2. Вебхук на NestJS (вместо long polling в проде)

  1. Поднять публичный HTTPS endpoint, например POST /webhooks/telegram (тот же Nest, что API).
  2. Вызвать метод Bot API: setWebhook с URL вебхука. В новых версиях API рекомендуют параметр secret_token — случайная строка; Telegram пришлёт её в заголовке X-Telegram-Bot-Api-Secret-Token, ваш контроллер обязан сверять с конфигом (защита от посторонних POST).
  3. Тело запроса — JSON Update (объект с полем update_id и одним из: message, callback_query, pre_checkout_query, successful_payment и др.).
  4. Идемпотентность: сохранять обработанные update_id в Redis или БД (TTL сутки+), чтобы при повторной доставке не задвоить действия.

3. Модуль в apps/api

  • TelegramModule: контроллер вебхука + сервис разбора Update.
  • Сервис вызывает существующие сервисы домена: BookingsService, EscrowService, уведомления — не дублировать бизнес-логику внутри «if telegram».
  • Для исходящих сообщений — HTTP POST https://api.telegram.org/bot<TOKEN>/sendMessage (или обёртка-библиотека). Лимиты Telegram: не спамить, учитывать FAQ.

4. Привязка аккаунта сайта к Telegram

Без привязки вы не знаете, какому users.id соответствует message.from.id.

  1. В БД: поле telegram_chat_id / telegram_user_id у пользователя или отдельная таблица связей.
  2. Deep link: в личном кабинете на сайте кнопка «Подключить Telegram» → генерируете одноразовый токен (Redis, 10–15 мин TTL) → ссылка https://t.me/YourBot?start=link_<token>.
  3. Бот на /start link_… вызывает API: обмен токена на привязку telegram_user_id ↔ текущий залогиненный пользователь (токен валидируется на сервере).
  4. Дальше все уведомления шлёте на сохранённый chat_id (для личных чатов часто совпадает с user id, для групп — другой id).

5. Сценарий вокруг эскроу (модель A — рекомендуемый старт)

  1. Событие «создана бронь / создан escrow pending_funding» → очередь или сразу sendMessage: текст + кнопка «Оплатить» с URL на страницу оплаты Next.js с bookingId (пользователь уже залогинен на сайте или по магической ссылке с подписанным JWT).
  2. После вебхука банка и перехода в funded → бот шлёт «Оплата получена, средства на эскроу».
  3. Менеджеру (отдельная привязка или группа): «Нужно подтвердить встречу» + inline-кнопки callback_data с коротким подписанным payload (confirm:bookingId:…), обработка callback_query → вызов вашего POST /escrow/.../confirm (с проверкой прав по telegram_user_id ↔ роль manager в БД).
  4. После released → уведомить клиента и при необходимости модель (если у них есть бот-привязка).

6. Telegram Payments и эскроу

Если используете sendInvoice / answerPreCheckoutQuery / обработку successful_payment:

  • pre_checkout_query — последняя проверка перед списанием: сумма, валюта, наличие брони, не отменена ли сделка. Ответ ok: true/false.
  • successful_payment — факт успешной оплаты в Telegram; здесь вы ставите у себя в системе то же, что после вебхука эквайринга: перевод эскроу к funded (если провайдер не шлёт отдельный вебхук — уточнить у провайдера; иногда дублируют).
  • Эскроу «до встречи» внутри Telegram не существует: логика холда/возврата остаётся у платёжного провайдера и вашего бэкенда.

7. Mini App (Telegram Web App)

  1. В BotFather привязать домен Mini App, получить данные для initData.
  2. Фронт внутри WebView открывает ваш URL; клиент передаёт initData (подпись Telegram) — на сервере проверяете подпись по алгоритму из документации, извлекаете user.
  3. Дальше мини-приложение вызывает ваши же REST эндпоинты (CORS и домены настроить). Оплата — iframe/редирект на эквайринг или встроенный виджет провайдера.
