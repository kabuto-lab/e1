Эпики и анализ Гант (вкладка в blueprint) Blueprint ролей

План на 12 рабочих недель плюс буфер. Сроки — оценка для одного активного потока разработки; при параллельных задачах недели могут сдвигаться. Недели не привязаны к календарным датам — стартуйте с «недели 1» от фиксации baseline.

A — API и безопасность B — Дашборд и данные C — Публичный UX D — Брони и деньги E — Качество и документы

Фаза 1 — недели 1–4

Неделя 1 A · E
  • API: ревизия models.controller — включить JwtAuthGuard на всех непубличных маршрутах; список публичных GET зафиксировать в комментарии/Swagger.
  • CI: pipeline: lint + tsc (или npm run build web/api) на каждый PR.
Неделя 2 A
  • Контракт: черновик: что отдаёт публичный каталог и /models/:slug — один «канонический» слой (согласование models vs profiles).
  • Тесты: 1–2 integration-теста: логин → запрос защищённого эндпоинта с валидным JWT.
Неделя 3 B
  • Дашборд: фильтрация пунктов бокового меню по роли пользователя (admin / manager / model / client).
  • Решение по заглушкам: «Клиенты» и «Финансы» — либо скрыть до готовности, либо завести минимальные страницы-заготовки без #.
Неделя 4 B
  • Бронирования: страница дашборда читает список из API (замена моков); отображение статусов из БД, без сложных действий при необходимости.
  • Проверка: ручной проход сценариев под каждой ролью после изменения меню.
Контрольная точка после недели 4 Закрыты очевидные дыры в auth на API; меню дашборда соответствует ролям; брони отображаются из бэкенда (хотя бы read-only).

Фаза 2 — недели 5–8

Неделя 5 B
  • Бронирования: действия менеджера/админа — смена статуса через API с валидацией переходов.
  • Модерация: список очереди из медиа с moderationStatus = pending (или агрегат по сущностям).
Неделя 6 B
  • Модерация: approve / reject с записью в БД и обновлением UI списка.
  • Клиенты (опционально): первая итерация списка клиентов из API для staff-ролей.
Неделя 7 C
  • Публичный профиль: компонент FadeSlider — только медиа с isPublicVisible, интервал и плавный кроссфейд.
  • Производительность: lazy-load изображений слайдера.
Неделя 8 C
  • Визуал: полировка героя/фона публичного профиля без WebGL-оверлея (water shader не в apps/web; архив в NON_PROJECT).
  • SEO: базовые Open Graph / title / description для страницы модели.
Контрольная точка после недели 8 Модерация контента замкнута на API; публичный профиль с слайдером и улучшенным первым впечатлением (статичный/лёгкий фон без обязательного WebGL).

Фаза 3 — недели 9–12

Неделя 9 C
  • Галерея: лайтбокс или полноэкранный просмотр на публичной анкете.
  • Дизайн: пройтись по каталогу и профилю по чеклисту из DESIGN.md (сетка, типографика).
Неделя 10 D
  • Бронь: сценарий создания запроса на бронирование со стороны клиента (форма → API) при готовности бэкенда.
  • API: зафиксировать машину состояний брони в сервисе (допустимые переходы, ошибки).
Неделя 11 D
  • Эскроу MVP: либо интеграция с выбранным провайдером (spike + один happy-path), либо задокументированный ручной эскроу в UI + поля статуса в БД.
  • Связка: отображение статуса оплаты/эскроу в карточке брони в дашборде.
Неделя 12 D · E
  • Уведомления: цепочка от события брони/модерации к e-mail (staging; конфиг под prod).
  • Документы: обновить CLAUDE.md и COMPREHENSIVE_AUDIT_AND_PLAN.md под фактическое состояние после цикла.
Контрольная точка после недели 12 Сквозной сценарий брони согласован с UI; эскроу в режиме MVP; публичная часть и дашборд согласованы с документацией.

Буфер и дальше
