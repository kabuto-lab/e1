### 2.3 TON USDT Escrow — production-grade

**Schema (`packages/db/src/schema/escrow*.ts` + migration `0009_ton_escrow_audit.sql`):**
- `escrow_transactions.paymentProvider ∈ {yookassa, cryptomus, manual, ton_usdt}`, BigInt `expectedAmountAtomic` / `receivedAmountAtomic`, UNIQUE indexes on `expected_memo`, `funded_tx_hash`, `release_tx_hash`, `refund_tx_hash`. Status enum: `pending_funding | funded | hold_period | released | refunded | disputed_hold | partially_refunded`.
- `escrow_ton_deposits`: idempotency by `tx_hash` UNIQUE, `logical_time` BigInt, JSONB `raw_payload` (TonAPI event snapshot).
- `escrow_audit_events`: `event_type`, `actor_type`, `correlation_id`, JSONB payload — **separate from `booking_audit_logs`** (good).

**Service (`apps/api/src/escrow/ton-escrow.service.ts`, ~660 lines):**
- `createIntent()` — env check, booking ownership, status gate, deterministic UNIQUE memo from escrowId.
- `recordDeposit()` — full `tonRepo.withTransaction`, memo match, **idempotent by tx_hash**, partial funding tracked, auto-transitions booking on exact match.
- `confirmRelease/Refund()` — strict state-machine guards via `RELEASABLE_ESCROW_STATUS` / `REFUNDABLE_ESCROW_STATUS` sets.
- `broadcastRelease/Refund()` — signs jetton transfer via `TonHotWalletService` (@ton/ton `WalletContractV4` + `mnemonicToPrivateKey`), records tx hash.
- All money is `BigInt`. All state changes are in transactions. Audit events append on every transition.

**Controller (`escrow.controller.ts`):** 7 TON endpoints; `POST /escrow/ton/deposit` is gated by `TonEscrowDepositGuard` (timing-safe compare of `x-ton-escrow-ingest` header, min-16-char secret). Legacy fiat endpoints kept for backwards compat.

**Indexer (`ton-escrow-indexer.service.ts`):** `OnModuleInit` polling gated by `TON_INDEXER_ENABLED=true`, configurable interval, `lastLt` tracking, bootstrap.

**Tests:** `ton-escrow.service.spec.ts` (viewer happy path, 403, 404; value objects `ton-address`, `escrow-memo`). **Unit only — no e2e yet.**

## 4. Risks & concerns

1. **`packages/db/drizzle/0009_ton_escrow_audit.sql` is MODIFIED in working tree.** If the original already ran on the VPS (`vps:after-pull` → `drizzle-kit push`), these edits will never apply. Either:
   - (a) revert the file and add a new `0010_*.sql` migration with the delta, or
   - (b) verify VPS did **not** yet apply 0009 and push current version.
2. **No e2e coverage for escrow.** Happy path `intent → deposit ingest → funded → release/refund` has only unit tests. The highest-value negative cases — double-spend, wrong-memo, amount mismatch, replay — are not in CI.
3. **Indexer single point of failure.** `TON_INDEXER_ENABLED` polling has no dead-man's-switch or alert if TonAPI goes silent. Add a staleness metric (`lastLt` age) and a panic log after N minutes without events.
4. **Hot wallet key custody.** Mnemonic lives in env (`TON_HOT_WALLET_MNEMONIC`). OK for MVP, but document rotation + backup and consider a signer service boundary before fiat volume.
5. **`dashboard/bookings` serving mock data** — if a manager opens it during demo, confusion guaranteed. High-visibility fix.
6. **Legacy fiat endpoints still live on escrow controller** with no provider wired. Either mark deprecated in Swagger or return 501 to avoid confusion.
7. **Docs drift itself is a risk** — contributors (including AI assistants) read `START_HERE.md` and `CAP.md` and make decisions on stale info. Either update or clearly demote them.
