### 2.4 TLA-режим: триггер, уровни, gates

Полная цитата ENTITY.md:142–177 (см. Приложение A для контекста):

    ## 9. TLA Entity — Triple-Level Architect (нетривиальный NestJS / TypeScript)

    **Когда включать:** задачи вроде нового модуля, эскроу, платежи, крупные
    изменения схемы БД, тесная связка с **`tickets`** и т.п. Тривиальные правки
    (опечатка, один guard, мелкий фикс) — без обязательного прохождения всех
    уровней.

    **Триггер от человека:** фраза **`TLA Entity, new task: [описание]`** —
    ассистент входит в режим и **не пропускает уровни**, пока человек явно
    не скажет перейти (например «Level 2» или «продолжай с уровня 3»).

    **Обязательное начало ответа ассистента в этом режиме:**

    > Understood. Entering Three-Level Architect mode.

    Далее сразу **Level 1**, если человек не указал иной уровень.

    ### Строгие правила (не нарушать)

    1. **Только монорепо:** `apps/api` (NestJS 10), `packages/db`
       (Drizzle + PostgreSQL), существующие модули (`users`, `tickets`,
       `profiles`, …). Стиль, имена и архитектура — как в текущем коде.

    2. **Три уровня — по порядку, без реализации «всего сразу»:**

       **Level 1 — Strategic planning**
       Понять требования, предложить высокоуровневую архитектуру, ключевые
       решения, точки интеграции (в т.ч. с `tickets`). Выход: план + вопросы,
       если неясно. **Кода нет.**

       **Level 2 — Architectural design**
       Полная проекция Drizzle-схемы (ограничения, индексы, связи), структура
       папок под модуль (например `apps/api/src/escrow/`), сущности домена,
       value objects, enums, интерфейсы репозиториев, при необходимости CQRS
       (команды/обработчики). TypeScript-интерфейсы и типы.
       **Полную бизнес-логику не писать.**

       **Level 3 — Incremental implementation**
       Реализация **по одному файлу за шаг**; перед следующим файлом —
       **согласование с человеком** (в этом режиме это ожидаемо).
       Продакшен-качество, комментарии по делу, существующие паттерны (Zod,
       Drizzle, DI, обработка ошибок). Для денег: транзакции, идемпотентность,
       аудит, безопасность.

    3. **Дополнительно:**
       - Деньги: **не** `Number` для денежных сумм → **BigInt** и/или value objects.
       - Изменения в БД — в **транзакциях**, где уместно.
       - Не ломать существующее поведение, особенно модуль **`tickets`**.
       - Для крипто-контура в документах проекта ориентир: **TON USDT** как
         основная сеть, где это зафиксировано в ТЗ/роадмапе.
       - Всегда учитывать безопасность, идемпотентность и аудит для платёжных
         и смежных потоков.

    **Согласование с другими правилами:** если в чате включён режим TLA по
    триггеру выше, шаг «подтверждение перед следующим файлом» на **Level 3**
    имеет приоритет над общим ускорением «без пауз». Вне режима TLA — обычный
    темп из §2.

### 7.2 Подводные камни (из доков)

ENTITY.md:85
    Пароль в Docker-томе задаётся при **первой** инициализации volume;
    смена строки в `.env` **сама по себе** не меняет уже созданный том
    → классическая ошибка **`password authentication failed for user "postgres"`**
    (часто **28P01**).

ENTITY.md:97–98
    API не подключается к PostgreSQL (**часто 28P01**) → Nest отдаёт
    **503** с сообщением про пароль. Каталог `/models` и дашборд показывают
    одну и ту же беду с разным UI.

ENTITY.md:102–103
    1. Пароль в Docker-томе задаётся при ПЕРВОМ создании volume; смена
       POSTGRES_PASSWORD в compose НЕ обновляет существующий том →
       расхождение с DATABASE_URL в ~/e1/.env. В .env держите
       POSTGRES_PASSWORD и пароль в DATABASE_URL одинаковыми (см. .env.example);
       docker-compose.dev.yml берёт POSTGRES_PASSWORD из .env.
       Не делайте docker compose down -v на проде — том БД сотрётся.
    2. pm2 restart escort-api сохраняет СТАРЫЙ process.env. После .env нужен
       startOrReload: npm run pm2:reload-api (см. ecosystem.config.cjs).

Короткий DevOps-блок для передачи (ENTITY.md:94):
    Senior DevOps-aware rule: single source of truth for docker-compose.dev.yml,
    .env.example, and VPS docs; keep POSTGRES_PASSWORD and DATABASE_URL in
    lockstep; after credential changes document volume recreation vs ALTER USER;
    on VPS prefer vps:after-pull + pm2:reload-api; always flag Postgres volume
    / password mismatch.

### 7.3 Что считается shared-state / необратимым

  - Запрет docker compose down -v на проде (ENTITY.md:91, 102)

не проектные документы.

Команды-якоря безопасной работы (ENTITY.md:117–122):

## 6. VPS (`~/e1`) — выкатка без «двух ошибок» (503 каталог / пароль БД)

### Единый контур: локально ↔ GitHub ↔ VPS

**Цель:** одна согласованная история про Docker, Postgres и env — без «работает только у меня» и без сюрпризов после `git pull`.

1. **Источники правды по инфраструктуре**
   - Локальная разработка: **`docker-compose.dev.yml`** в корне репозитория (имя проекта `escort-dev`, контейнер Postgres **`escort-postgres`** и т.д.).
   - Продакшен на VPS: фактический compose и сеть — как задеплоено на сервере; текстовый регламент — **`docs/DEPLOY_SERVER.md`**. Отдельного **`docker-compose.prod.yml` в корне репо может не быть** — не придумывать файлы; при необходимости выносить prod-фрагменты в репозиторий осознанным PR.
   - Пример переменных для людей и ИИ: **`.env.example`** — держать в актуальном виде вместе с любыми изменениями контракта (Postgres, URL БД, порты).

2. **Синхронизация Postgres (обязательно)**
   - **`POSTGRES_PASSWORD`** (или эквивалент в compose) и пароль в **`DATABASE_URL`** должны **совпадать** с тем, что реально ожидает роль в БД.
   - Пароль в Docker-томе задаётся при **первой** инициализации volume; смена строки в `.env` **сама по себе** не меняет уже созданный том → классическая ошибка **`password authentication failed for user "postgres"`** (часто **28P01**).
   - После смены креденшелов в репозитории в инструкциях к PR/деплою всегда предусматривать либо **выравнивание пароля** (`ensure:database`, `ALTER USER`, `db:align-password`), либо **осознанное пересоздание тома** (только dev/staging или с бэкапом — **не** как дефолт для прода).

3. **Поведение ИИ при задачах на backend / БД / Docker / деплой / env / escrow / bookings / платежи**
   - Явно проверять согласованность **compose ↔ `.env.example` ↔ `DATABASE_URL`**.
   - Не предлагать конфиг, который зависит от «секрета только на VPS» без документирования в **`DEPLOY_SERVER.md`** или `.env.example` (плейсхолдер + комментарий).
   - В ответах про деплой **по умолчанию** опираться на **`npm run vps:after-pull`** из `~/e1`; **`docker volume rm`** / **`docker compose down -v`** упоминать только как вариант **с потерей данных** или для чистого dev/staging — см. таблицу регламента ниже.

**Короткий блок для вставки в другой чат (EN):**
*Senior DevOps-aware rule: single source of truth for `docker-compose.dev.yml`, `.env.example`, and VPS docs; keep `POSTGRES_PASSWORD` and `DATABASE_URL` in lockstep; after credential changes document volume recreation vs `ALTER USER`; on VPS prefer `vps:after-pull` + `pm2:reload-api`; always flag Postgres volume / password mismatch.*

### Одна причина двух симптомов

API не подключается к PostgreSQL (**часто 28P01**) → Nest отдаёт **503** с сообщением про пароль. Каталог `/models` и дашборд показывают одну и ту же беду с разным UI.

### Почему ломается после pull

1. **Пароль в Docker-томе** задаётся при **первом** создании volume; смена `POSTGRES_PASSWORD` в compose **не** обновляет существующий том → расхождение с `DATABASE_URL` в `~/e1/.env`. В `.env` держите **`POSTGRES_PASSWORD` и пароль в `DATABASE_URL` одинаковыми** (см. `.env.example`); `docker-compose.dev.yml` берёт `POSTGRES_PASSWORD` из `.env`. Не делайте **`docker compose down -v`** на проде — том БД сотрётся.
2. **`pm2 restart escort-api`** сохраняет **старый** `process.env`. После `.env` нужен **`startOrReload`**: **`npm run pm2:reload-api`** (см. `ecosystem.config.cjs`).

### Регламент

| Когда | Что сделать |
|--------|-------------|
| Закончил правки локально | **commit + `git push`** — иначе VPS не увидит скрипты и фиксы |
| На VPS после `git pull` | Из `~/e1`: **`npm run vps:after-pull`** (`npm ci` → `build` → **`check:postgres-env`** → **`ensure:database`** → **`db:migrate`** → **`pm2:reload-api`**) |
| Новый `next build` на VPS | При процессе **`escort-web`** в PM2: **`pm2 reload escort-web`** (смена env для веба: **`--update-env`**) |
| Другой контейнер Postgres | В `.env`: **`POSTGRES_CONTAINER=имя`** |
| Postgres не в Docker | Вручную **`ALTER USER`** под пароль из `DATABASE_URL`, затем **`npm run pm2:reload-api`** |

### Команды-якоря

- `npm run vps:after-pull` — основной сценарий обновления на сервере
- `npm run db:migrate` — применение SQL-миграций Drizzle (входит в `vps:after-pull`; при ошибке «column does not exist» после деплоя часто не прогнали миграции)
- `npm run check:postgres-env` — проверка, что `POSTGRES_PASSWORD` и пароль в `DATABASE_URL` совпадают
- `npm run ensure:database` — проверка + при Docker и 28P01 авто-`ALTER ROLE` под `.env`
- `npm run db:align-password` — только выравнивание пароля роли в контейнере
- `npm run verify:database` — проверка строки подключения; вызывается из `scripts/start-api-prod.mjs` перед стартом API

### Правило для ИИ в диалоге по SSH

Выдавать **строго по одной shell-команде** за сообщение (если пользователь не просит иначе); следующую — после результата. Типовая цепочка по смыслу: `git pull` → при необходимости `npm run vps:after-pull` (не список из пяти команд одним блоком).

## Aesthetic Direction

- **Direction:** Luxury/Refined — Modern luxury with restrained elegance
- **Decoration level:** Intentional (subtle grain texture, minimal ornamentation)
- **Mood:** Sophisticated, exclusive, intimate without being explicit. Wealth whispers.
- **Reference sites:** High-end hotel lobbies, private members clubs, luxury watch brands

### Design Principles

1. **Restraint over excess** — Gold is rare and meaningful, used only for what matters
2. **Warmth over coldness** — Grain texture, warm gold tones, not sterile flat design
3. **Structure over chaos** — Grid-disciplined layouts, predictable alignment
4. **Clarity over cleverness** — Typography first, decoration second

## Typography

### Font Stack (Google Fonts — Free, Fast)

| Role | Font | Weights | Usage |
|------|------|---------|-------|
| **Display/Hero** | Unbounded | 400, 500, 600, 700, 800, 900 | Page titles, hero headings, model names, logo |
| **Body** | Inter | 300, 400, 500, 600, 700 | Body copy, descriptions, labels, UI text |
| **Data/Tables** | Inter (tabular-nums) | 400, 500 | Prices, stats, dates |

### Why These Fonts

- **Unbounded:** Geometric, rounded, distinctive display face — premium feel without being stuffy. Excellent for hero text and brand marks. Very recognizable at large sizes.
- **Inter:** Industry-standard UI font. Extremely readable at all sizes, excellent tabular figures, extensive weight range. Clean and professional.

## Motion

### Approach: Intentional

Only animate when it aids comprehension. No gratuitous motion.

### Easing

| Name | Value | Usage |
|------|-------|-------|
| Enter | `cubic-bezier(0.25, 0.1, 0.25, 1)` | Smooth ease-out for entrances |
| Exit | `cubic-bezier(0.25, 0.1, 0.25, 1)` | Smooth ease-in for exits |
| Move | `cubic-bezier(0.4, 0, 0.2, 1)` | Smooth ease-in-out for transitions |

### Duration

| Name | Range | Usage |
|------|-------|-------|
| Micro | 50-100ms | Hover states, toggles, switches |
| Short | 150-250ms | Button presses, small transitions |
| Medium | 250-400ms | Card hovers, panel slides |
| Long | 400-700ms | Page transitions, modals, fade slider |

### CSS Variables

```css
:root {
  --ease-enter: cubic-bezier(0.25, 0.1, 0.25, 1);
  --ease-exit: cubic-bezier(0.25, 0.1, 0.25, 1);
  --ease-move: cubic-bezier(0.4, 0, 0.2, 1);
  --duration-micro: 100ms;
  --duration-short: 200ms;
  --duration-medium: 300ms;
  --duration-long: 500ms;
}
```

### Principles

1. **Respect `prefers-reduced-motion`** — Disable non-essential animation
2. **Purpose over decoration** — Animate to aid understanding, not to impress
3. **Performance first** — Use `transform` and `opacity` only (GPU-accelerated)

function resolveEnvFilePath(): string {
  const cwd = process.cwd();
  for (let depth = 0; depth < 8; depth++) {
    const base = depth === 0 ? cwd : resolve(cwd, ...Array(depth).fill('..'));
    const envPath = resolve(base, '.env');
    if (existsSync(envPath)) return envPath;
  }
  return resolve(cwd, '.env');
}
