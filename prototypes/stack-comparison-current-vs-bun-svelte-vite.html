Сравнение технологического стека

Текущая реализация платформы Lovnge (монорепозиторий в репозитории) против целевого набора Bun + Svelte + TypeScript + Vite. Сравнение носит архитектурный характер: при миграции фронта на Svelte+Vite для паритета с Next.js обычно добавляют SvelteKit (SSR, роутинг, формы) — это отражено в ячейках.

Текущий стек (референс) Bun + Svelte + TS + Vite (и типичные соседи)

Основная таблица сравнения

Слой / аспект Текущий стек Bun + Svelte + TypeScript + Vite
Язык TypeScript (строгий в пакете @escort/db; API с частично смягчённым strict) TypeScript — тот же язык; единые tsconfig для фронта и при желании для API на Bun
Среда выполнения JS Node.js ≥ 22 Bun — другой рантайм и менеджер пакетов; совместимость с большинством npm-пакетов высокая, но нативные аддоны и краевые случаи Nest/Next требуют проверки
Пакеты / установка npm workspaces, package-lock Часто bun install, lockfile Bun; можно смешивать с npm при необходимости
Монорепозиторий Turborepo + workspaces (apps/*, packages/*) Возможны те же workspaces + Turbo, либо bun workspaces, либо Nx; Turbo не привязан к Node
Фреймворк UI (гость + дашборд) Next.js 15 (App Router), React 19, RSC/SSR по маршрутам Svelte (компоненты). «Голый» Vite + Svelte = в основном SPA; для SSR/SEO как у Next — SvelteKit (Vite внутри)
Сборка фронтенда Next/SWC (и webpack-настройки под Windows) Vite — быстрый dev HMR, явный контроль над бандлом; прод-сборка через Rollup (Vite)
Стили Tailwind CSS, PostCSS Tailwind + Vite/SvelteKit — стандартная связка; плюс scoped CSS Svelte
Backend HTTP API NestJS 10 (Express), модули, DI, Swagger, guards, pipes Варианты: Elysia / Hono на Bun; или оставить Nest под Bun (экспериментально); или логика в SvelteKit server (+hooks, form actions) — иная архитектура, без 1:1 модулей Nest
Контракт API REST JSON, OpenAPI в рантайме (/api/docs) Тот же REST + tRPC/OpenAPI при желании; Swagger «из коробки» как у Nest — отдельная настройка
ORM / доступ к БД Drizzle ORM + драйвер postgres, схемы в packages/db Drizzle официально дружит с Bun; можно перенести схемы с минимальными правками драйвера/пула
База данных PostgreSQL 16 Без изменений по сути той же СУБД
Аутентификация JWT (access+refresh), Passport, bcrypt, email SHA-256 hash в БД Та же модель на любом фреймворке; нужно переписать guards/interceptors → middleware/handlers
Файловое хранилище S3 API (@aws-sdk) + MinIO, presigned PUT Те же SDK или @aws-sdk под Bun; проверить совместимость стримов для больших файлов
Кэш / очереди Redis (env обязателен); throttler в памяти Redis без привязки к стеку; клиент на Bun — выбрать библиотеку (ioredis и т.д.)
Прокси API с сайта Next rewrites /api/* → Nest (dev) или прямой NEXT_PUBLIC_API_URL SvelteKit: hooks.server proxy, или только CORS + прямой URL в браузере
Валидация env Zod при старте Nest (validation.schema.ts) Тот же Zod в точке входа Bun/Elysia/SvelteKit
Тесты Jest (Nest), типовые e2e Vitest (родственник Vite), Playwright для e2e — часто удобнее единообразия с Vite
Продакшен-процесс Сборка turbo run build, API node …/main.js, Web next start, PM2 Vite/SvelteKit build → статика + Node/Bun adapter; или Docker; PM2 с bun run / node
Объём переписывания — Высокий для UI (все .tsx → Svelte). Backend можно оставить на Nest+Node отдельно и только сменить фронт — тогда Bun не обязателен для API

Дополнительно: сильные стороны

Критерий Текущий стек Bun + Svelte + TS + Vite
Экосистема UI Огромная экосистема React, готовые паттерны для админок Меньше пакетов «из коробки», зато меньше runtime и часто проще реактивная модель
SEO / SSR Next — зрелая модель SSR/SSG, кэширование маршрутов SvelteKit на том же уровне задач; чистый Vite SPA — хуже для SEO без отдельного SSR-слоя
Скорость dev Хорошая; на Windows в проекте есть обходы для webpack/SWC Vite HMR очень быстрый; Bun ускоряет install и скрипты
Предсказуемость продакшена Node + Nest + Next — десятилетия практики на VPS Bun быстрее эволюционирует; для критичного продакшена иногда держат API на Node до стабилизации

Итоговая формулировка

  • TypeScript в обоих вариантах — общий знаменатель; перенос типов и Drizzle-схем реалистичен.
  • Svelte + Vite заменяют React + Next не построчно: нужен план по роутингу, SSR, данным (load functions в SvelteKit vs RSC/server components в Next).
  • Bun — замена Node для скриптов и возможно API; полная замена без регрессий требует чек-листа по нативным модулям (bcrypt часто ок, но проверять).
  • Прагматичный гибрид: оставить Nest+Postgres, переписать только фронт на SvelteKit + Vite — меньший риск, чем одновременно менять рантайм и оба слоя.
