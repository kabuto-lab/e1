Скорость сайта: от улучшений к «сумасшедшему» стеку

Варианты значительно ускорить работу сайта — по слоям: сеть, HTML, JS, данные, медиа. Третья колонка — не «обязательный чеклист», а референс максимальной скорости (часто дороже в разработке и эксплуатации). Метрики: TTFB (первый байт), LCP (крупнейший контент), INP (отзывчивость), CLS (стабильность вёрстки).

Базовый / типичный путь Сильное ускорение без полной смены парадигмы Экстремум скорости (edge, минимум JS, агрессивный кэш)

Таблица 1. По слоям: что ускоряет и насколько радикально

Слой База (часто уже есть) Сильный выигрыш «Сумасшедшая» скорость
Сеть / TLS Один регион VPS, HTTP/2 CDN перед статикой и медиа; Brotli; preconnect к API и шрифтам; HTTP/3 на CDN Edge PoP у пользователя (Workers/Pages + глобальный anycast); кэш на границе по правилам; минимизация кросс-доменных запросов
HTML документ SSR на каждый запрос с холодным стартом SSG/ISR для публичных страниц; stale-while-revalidate; сегменты кэша по Cookie только где нужно Статический shell + edge-кэш HTML; персонализация островками/клиентом; streaming HTML первыми байтами
JavaScript в браузере Тяжёлый SPA, весь бандл на первом экране Code splitting по маршрутам; динамический import; удалить неиспользуемые полифилы; серверные компоненты / меньше гидрации Островная архитектура (Astro), resumability (Qwik), или почти нулевой JS на лендингах; критический CSS inline
Шрифты Полные веса Google Fonts блокируют рендер Self-host + subset + font-display: swap; variable font одним файлом Только системный стек на части страниц; или WOFF2 + строгий preload только для above-the-fold
Изображения PNG/JPEG без размеров, без lazy next/image или аналог; AVIF/WebP; явные width/height; lazy ниже сгиба Отдельный image CDN (imgproxy, Cloudflare Images, Cloudinary); адаптивные srcset; LQIP/blur; приоритет только для LCP
API / БД Каждый запрос — N запросов в Postgres без пула PgBouncer/пул; индексы и EXPLAIN; Redis для горячих read; батчинг запросов Read replica рядом с API; edge-friendly Postgres (Neon + Hyperdrive / аналоги) для низкой латентности от воркера; материализованные представления / CQRS для чтения
Кэш приложения Только in-memory на одном инстансе Redis для сессий и публичных списков с TTL; ETag/304 для API где уместно Многоуровневый: CDN → edge KV → Redis → БД; явные инвалидации по событию
Рантайм бэкенда Node + тяжёлый cold start в serverless Долгоживущие процессы (PM2); Bun для быстрых скриптов и иногда API Лёгкие edge handlers (Hono/Elysia на Workers) для read-only путей; тяжёлая логика — отдельный региональный сервис

Таблица 2. Референс стека «максимум скорости» (сборка под экстремальный LCP/TTFB)

Ориентир для зелёного Core Web Vitals при глобальной аудитории; не все пункты нужны одновременно.

Компонент Технологии / паттерн Зачем
Край сети Cloudflare (или Fastly) — CDN + Cache Rules + (опц.) Argo Smart Routing Ближайший PoP, кэш статики и части HTML, сжатие, TLS оптимизация
Статика и фронт Astro или SvelteKit + adapter-static где возможно; острова для интерактива Минимум JS на первом экране, быстрый TTI
Интерактив «без тяжести» Svelte 5 / Preact в островах; или Qwik при очень больших деревьях состояния Меньший runtime и быстрее гидрация/восстановление
Сборка Vite + Rollup; анализ бандла (rollup-plugin-visualizer) Контроль размера чанков, tree-shaking
Edge compute Cloudflare Workers + Hono или Elysia (Bun — на origin, не на CF Workers) Логика на миллисекундах от пользователя для read-путей
БД + латентность Neon / Supabase / managed Postgres + пул (Hyperdrive, PgBouncer); read replicas Не исчерпывать соединения; снизить RTT для чтения
Объекты / медиа R2 / S3 + CDN; отдельный pipeline ресайза (imgproxy, CF Images) Быстрая отдача; правильные форматы и размеры
Кэш данных Redis или Cloudflare KV/Durable Objects для горячих ключей Срезать повторные походы в БД на пике
Наблюдаемость RUM (web-vitals), трассировка TTFB по слоям (CDN → origin → DB) Ускорять измеренное узкое место, а не «всё сразу»

Таблица 3. Быстрые победы в текущем стеке (Next + Nest), если не менять фреймворки

Мера Что сделать Ожидаемый эффект
CDN для _next/static и публичных файлов Вынести статику за reverse-proxy с кэшем или использовать хостинг с edge (Vercel/Cloudflare) Меньше RTT для повторных визитов; стабильнее LCP для шрифтов/чанков
Кэш GET API Cache-Control + CDN для публичных списков; Redis на origin для тяжёлых агрегаций Снижение нагрузки и TTFB для каталогов
Оптимизация Postgres Пул соединений; недостающие индексы; убрать N+1 в сервисах Быстрее ответы Nest → ниже TTFB при SSR/proxy
Next: loading / streaming Suspense boundaries, loading.tsx, по возможности PPR (где поддерживается) Раньше видимый контент; лучше воспринимаемая скорость
Меньше JS в клиенте Server Components где уместно; не тянуть тяжёлые библиотеки на лендинги Лучше INP и Time to Interactive

Короткий вывод

  • Самый большой скачок обычно даёт связка CDN + правильный кэш HTML/API + лёгкий первый экран, а не смена языка фреймворка.
  • «Сумасшедший» стек почти всегда означает edge + минимальный клиентский JS + агрессивное кэширование чтения и более сложную модель инвалидации.
  • Для проекта в монорепо разумный путь: сначала таблица 3 и метрики в проде, потом точечно острова/edge для самых горячих страниц.
