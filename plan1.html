🎯 Обзор проекта

Миссия: Миграция с WordPress/JetEngine/WooCommerce на безопасную, масштабируемую архитектуру NestJS/NextJS/PostgreSQL, способную поддерживать зашифрованные сообщения в реальном времени, сложные state-машины эскроу и криптографическую анонимность.

Текущее состояние (WordPress) vs Целевое состояние

Возможность WordPress (текущее) Современный стек (целевое)
Чат в реальном времени ❌ Только AJAX-опрос ✅ WebSocket с E2E шифрованием
Маскировка телефона ❌ Открытый текст в wp_usermeta ✅ Токенизированное хранилище + виртуальные номера
Логика эскроу ❌ Немедленное списание ✅ Много состояний с заморозкой
Изоляция ролей ❌ WP_Roles (грубо) ✅ RBAC на уровне полей + RLS
Хранение данных ❌ Осколки meta при удалении ✅ Криптографическое уничтожение
Масштабируемость ❌ Process-per-request ✅ Асинхронный I/O, горизонтальное масштабирование

Правила реализации

  • НИКОГДА не хранить номера телефонов в открытом виде в логах
  • ВСЕГДА использовать UUID для идентификаторов (никаких последовательных ID)
  • ПРИМЕНЯТЬ Row-Level Security (RLS) ко ВСЕМ таблицам БД
  • РЕАЛИЗОВАТЬ прямую секретность для сообщений (новый ключ на сессию)
  • АУДИРОВАТЬ каждый доступ к данным через неизменяемое логирование
  • ВАЛИДИРОВАТЬ все переходы состояний в эскроу-машине
  • ШИФРОВАТЬ бэкапы отдельно от ключей приложения
  • ИЗОЛИРОВАТЬ доступ менеджеров (видят только свои модели)

Data Flow: Payment Processing

Primary Flow (YooKassa):

1. Client initiates booking → System creates payment intent (amount, hold=true)
2. Redirect to YooKassa widget → Client enters card details
3. YooKassa authorizes (holds) funds → Returns payment_id
4. Webhook: payment.waiting_for_capture → System transitions Escrow to FUNDED
5. 24h hold period → Auto-capture and release to model (minus 20% fee)
6. Webhook: payment.succeeded → Mark booking COMPLETED

Failure Scenarios:
- payment.canceled → Escrow → REFUNDED (no funds taken)
- refund.succeeded → Escrow → REFUNDED (funds returned)
- Dispute raised → Escrow → DISPUTED (manual admin review)
